# 发布中心与发布记录筛选整改方案

## 1. 任务背景

本方案用于指导 AI agent 后续落地发布中心与发布记录页的筛选逻辑整改。

当前问题不是单纯的 UI 文案问题，而是前端状态边界混用导致的交互歧义：

- 发布中心、发布记录页、创建发布单共用 `selection`。
- 发布中心 UI 有“全部发布”“全部服务”“全部环境”的表达，但数据加载后会自动把服务、环境等补成第一个可用对象。
- 发布记录页默认启用“仅当前服务与环境”，而“当前”来自全局状态或第一个可用对象，不一定是用户主动选择的上下文。
- 发布中心进入发布记录页时没有通过 URL 明确传参，只是两个页面共享同一份前端内存状态。
- 工作台也依赖同一套 `selected` 派生上下文；拆分状态时不能只处理创建页、发布中心和发布记录页。

整改目标是：发布中心默认能稳定查看全部发布单；发布记录页默认能查看全部执行记录；从发布详情等明确上下文进入时，才带入可见、可清除的筛选条件。

## 2. 当前代码事实

执行前必须先重新核对当前代码；以下是本方案制定时的代码锚点。

### 2.1 前端路由

- `web/src/App.tsx` 中 `Page` 包含 `releases`、`release-detail`、`deploys`、`create` 等页面。
- `routeFromLocation()` 只解析 path：
  - `/releases` -> 发布中心。
  - `/releases/new` -> 创建发布单。
  - `/releases/{id}` -> 发布详情。
  - `/deploys` -> 发布记录。
- `pathForPage()` 也只生成 path，没有 query 参数。

### 2.2 前端状态结构

- `Selection` 包含：
  - `serviceID`
  - `environmentID`
  - `versionID`
  - `targetID`
  - `userID`
- `ListFilters` 包含：
  - `scoped`
  - `releaseStatus`
  - `deployStatus`
- `selection` 当前同时服务于创建发布单、发布中心筛选、发布记录筛选和发布详情展示。
- `filters.scoped` 默认是 `true`，导致发布记录页默认按当前服务与环境过滤。
- `keepOrFirst()` 归一化会在数据变化后把空 `serviceID/environmentID/versionID/targetID/userID` 补成第一个可用对象。这是发布中心“全部”状态不稳定的直接根因。

### 2.3 发布中心筛选

发布中心的 `visibleReleases` 在前端内存中过滤：

- `filters.releaseStatus`
- `selection.serviceID`
- `selection.environmentID`
- `releaseView`
- `releaseSource`
- `releaseQuery`
- `releaseTimeRange`

服务和环境筛选逻辑本身支持空值代表全部，但 `selection` 会被自动补默认值，因此“全部”状态不稳定。

### 2.4 发布记录筛选

发布记录页的 `filteredDeploys` 在前端内存中过滤：

- `filters.deployStatus`
- `filters.scoped`
- 当前 `selected.service`
- 当前 `selected.environment`

`selected` 会基于 `selection` 查找实体，并在缺省时 fallback 到第一个服务和环境。因此发布记录页的“仅当前服务与环境”可能来自系统默认，而不是用户意图。

### 2.5 工作台上下文

- `selected` 是从 `selection` 和 `state` 派生的全局上下文。
- 工作台页面使用 `workbenchReleases` 展示“待我确认”“我发起的运行中”“我发起的最近失败”。
- `workbenchReleases` 当前依赖 `selected.user`，而 `selected.user` 优先使用 `currentUser`，其次才 fallback 到 `selection.userID` 或用户列表第一个。
- 拆分 `selection` 时，工作台必须保留基于登录态和可 fallback 的偏好上下文，否则会破坏工作台默认展示。

### 2.6 后端接口

当前路由：

- `GET /api/v1/release-requests`
- `GET /api/v1/deploy-records`
- `GET /api/v1/deploy-records/{id}/server-logs`

当前实现返回全量列表：

- `internal/httpapi/release.go` 的 `listReleases()` 调用 `service.List()`。
- `internal/app/release.go` 的 `List()` 调用 `store.ListReleaseRequests()`。
- `internal/repository/release.go` 的 `ListReleaseRequests()` 无查询参数，按 `created_at DESC, id DESC` 返回全部发布单。
- `internal/httpapi/deploy.go` 的 `listDeployRecords()` 调用 `store.ListDeployRecords()`。
- `internal/repository/deploy.go` 的 `ListDeployRecords()` 无查询参数，按 `created_at DESC, id DESC` 返回全部发布记录。

### 2.7 权限与分页

- 登录会话用户可读取发布单和发布记录。
- API key 读取发布单需要 `release:read`，读取发布记录需要 `deploy:read`。
- 当前没有行级权限过滤。
- 当前发布单列表和发布记录列表没有分页。

## 3. 产品目标

### 3.1 必须达成

- 发布中心默认展示全部发布单。
- 发布中心服务和环境筛选可清空，清空后不会被自动补回第一个服务或环境。
- 发布记录页默认展示全部发布记录。
- 从发布详情进入发布记录页时，明确按当前发布单过滤，并给出可见提示。
- 发布中心筛选、发布记录筛选、创建发布单选择互不污染。
- 刷新、浏览器返回、复制链接时，列表筛选语义稳定。
- 工作台的“我的发布”语义继续来自登录态 `currentUser`，不依赖列表筛选。

### 3.2 不在本次优先范围

- 不引入复杂 RBAC 或行级权限。
- 不改变发布单和发布记录的领域状态机。
- 不重构整个前端页面结构。
- 不做统计分析、趋势图、队列大盘等 P1 能力。
- 不在本次加入项目维度筛选；如后续需要按项目查看发布单，应单独设计 `project_id` 筛选及服务联动。

## 4. 设计原则

- 创建发布单需要明确对象，可以默认第一个可用服务、环境、版本、部署目标。
- 工作台是个人任务入口，可以保留基于登录态和偏好上下文的 fallback。
- 列表查询默认不应隐式选择第一个对象；空筛选必须表示全部。
- 只有用户明确选择上下文，或从详情页这类强上下文入口进入时，才带筛选。
- 被带入的筛选必须可见、可清除、可通过 URL 复现。
- 发布中心是发布单工作区；发布记录页是执行历史和日志定位页，两者不共享隐式状态。
- `currentUser` 是“我发起的”和工作台个人任务的唯一用户来源，不把 `userID` 塞回列表筛选。

## 5. 目标交互

### 5.1 发布中心

默认状态：

- 视图：全部发布。
- 服务：全部服务。
- 环境：全部环境。
- 状态：全部状态。
- 来源：全部来源。
- 时间：全部时间。
- 搜索：空。

用户可切换：

- 待我确认。
- 我发起的。
- 全部发布。

服务和环境筛选：

- 必须支持“全部”。
- 清空后保持空筛选，不再回填第一个服务或环境。
- 建议提供“清除筛选”按钮，一次清除服务、环境、状态、来源、时间、搜索。

进入发布记录：

- 顶部“发布记录”按钮默认进入全部发布记录。
- 本次不新增“按当前筛选查看记录”入口；如未来确有需要，再单独设计，避免发布中心出现两个相近的发布记录入口。

### 5.2 发布详情

发布详情仍然是强上下文页面。

关联发布记录区域点击记录时：

- 进入发布记录页。
- URL 应带 `release_request_id`。
- 发布记录页显示提示：`正在查看发布单 rel_xxx 的执行记录`。
- 提供“查看全部记录”按钮。

### 5.3 发布记录页

默认状态：

- 状态：全部状态。
- 发布单：全部。
- 服务：全部服务。
- 环境：全部环境。

筛选来源：

- 直接进入 `/deploys`：展示全部记录。
- 从发布详情进入：按 `release_request_id` 过滤。
- 从服务或环境上下文进入：可按 `service_id` / `environment_id` 过滤，但必须显示筛选提示。
- 本次默认只实现“直接进入全部”和“详情进入当前发布单”两类路径；服务或环境上下文入口留给后续需要时再接入。

提示条规则：

- 存在 `release_request_id` 时，提示当前发布单上下文。
- 存在 `service_id` 或 `environment_id` 时，提示当前服务/环境上下文。
- 提示条必须有“清除筛选”或“查看全部记录”动作。

### 5.4 工作台

工作台不是全量查询页，不使用发布中心或发布记录页筛选。

规则：

- “待我确认”“我发起的运行中”“我发起的最近失败”继续基于 `currentUser`。
- 工作台可以保留独立的偏好上下文，用于显示默认服务/环境或后续跳转预填。
- 工作台偏好上下文不得反向写入发布中心或发布记录页筛选。

## 6. 状态设计

将当前混用状态拆成四类。

### 6.1 创建发布单状态

建议命名：

```ts
type CreateSelection = {
  serviceID: string;
  environmentID: string;
  versionID: string;
  targetID: string;
};
```

用途：

- 仅用于 `/releases/new`。
- 允许 fallback 到第一个可用对象。
- 允许从部署目标创建成功、回滚、重新发布等流程预填。
- 当前 `keepOrFirst()` 归一化只能继续作用在 `createSelection` 这一类状态上，不能作用在发布中心或发布记录筛选上。

### 6.2 发布中心筛选状态

建议命名：

```ts
type ReleaseListFilters = {
  view: 'pending' | 'mine' | 'all';
  serviceID: string;
  environmentID: string;
  status: string;
  source: string;
  timeRange: string;
  query: string;
};
```

规则：

- 空 `serviceID` 表示全部服务。
- 空 `environmentID` 表示全部环境。
- `status/source/timeRange` 使用 `all` 表示全部。
- 不允许自动 fallback 到第一个服务或环境。
- `mine` 视图从 `currentUser.id` 解析，不读取 `selection.userID`，也不在 `ReleaseListFilters` 中增加 `userID`。

### 6.3 发布记录筛选状态

建议命名：

```ts
type DeployListFilters = {
  releaseRequestID: string;
  serviceID: string;
  environmentID: string;
  status: string;
};
```

规则：

- 空字段表示全部。
- `releaseRequestID` 优先级高于 `serviceID/environmentID`。
- 移除 `scoped: boolean`，不要保留第二套“仅当前服务与环境”真值源。
- 如 UI 仍需要快捷动作，只能表现为一次性填入或清空显式的 `serviceID/environmentID`。

### 6.4 工作台偏好上下文

建议命名：

```ts
type WorkbenchContext = {
  serviceID: string;
  environmentID: string;
  userID: string;
};
```

规则：

- `userID` 优先来自 `currentUser.id`。
- 可 fallback 到第一个服务或环境，用于工作台摘要或后续跳转预填。
- 不参与发布中心和发布记录页列表过滤。
- 不写入发布中心和发布记录页 URL query。

### 6.5 归一化拆分要求

必须拆掉当前单一的 `keepOrFirst()` 归一化路径：

- `createSelection`：保留 `keepOrFirst()` 或等价 fallback。
- `workbenchContext`：可保留服务/环境 fallback，用户必须优先 `currentUser`。
- `releaseListFilters`：不进入 `keepOrFirst()`，空值必须保持空值。
- `deployListFilters`：不进入 `keepOrFirst()`，空值必须保持空值。

## 7. URL 设计

### 7.1 发布中心

建议 URL：

```text
/releases?view=all&status=all&source=all&time_range=all
/releases?view=mine&service_id=svc_xxx&environment_id=env_xxx&status=running
```

解析规则：

- 缺失 query 时使用默认全部。
- 空 `service_id` / `environment_id` 表示全部。
- `time_range` 对应前端 `timeRange` 字段。
- URL 中只保留有意义的参数，避免产生噪声。

### 7.2 发布记录

建议 URL：

```text
/deploys
/deploys?release_request_id=rel_xxx
/deploys?service_id=svc_xxx&environment_id=env_xxx&status=failed
```

解析规则：

- `/deploys` 展示全部发布记录。
- `release_request_id` 存在时优先按发布单过滤。
- `release_request_id` 不存在时，才使用服务、环境、状态筛选。

## 8. 接口设计

### 8.1 短期实现

短期可以继续使用当前全量接口，并在前端按拆分后的筛选状态过滤。

前端整改优先级高于后端接口扩展，因为当前用户痛点主要来自状态混用和默认筛选。

### 8.2 中期接口扩展

当发布单和发布记录数据量增长后，再增加服务端筛选与分页。

发布单列表：

```text
GET /api/v1/release-requests?service_id=&environment_id=&status=&source=&created_by_id=&q=&created_from=&created_to=&limit=&offset=
```

发布记录列表：

```text
GET /api/v1/deploy-records?release_request_id=&service_id=&environment_id=&status=&limit=&offset=
```

语义：

- 空参数表示不筛选。
- 默认排序保持 `created_at DESC, id DESC`。
- 默认 `limit` 建议 50。
- `deploy-records` 按 `service_id/environment_id` 筛选时可 join `release_requests`。

## 9. 建议实施步骤

### 阶段 1：拆分前端状态

目标：

- 创建发布单、工作台、发布中心、发布记录四类状态互不污染。

建议步骤：

1. 新增 `CreateSelection`、`WorkbenchContext`、`ReleaseListFilters`、`DeployListFilters`。
2. 将创建页读取和修改逻辑迁移到 `createSelection`。
3. 将工作台所需的服务/环境/用户默认上下文迁移到 `workbenchContext`，其中用户优先来自 `currentUser`。
4. 将发布中心 `visibleReleases` 迁移到 `releaseListFilters`，`mine` 视图直接使用 `currentUser.id`。
5. 将发布记录 `filteredDeploys` 迁移到 `deployListFilters`，并移除 `scoped`。
6. 拆分 `keepOrFirst()` 归一化：只让 `createSelection` 和 `workbenchContext` 参与 fallback，列表筛选保持空值。
7. 保留发布详情自己的 active release / active deploy 状态。

验收：

- 发布中心默认全部服务、全部环境。
- 创建发布单仍能默认选中第一个可用服务、环境、版本、部署目标。
- 发布记录页默认全部记录。
- 工作台仍能展示当前登录用户相关任务。

### 阶段 2：修正跳转语义和最小 URL 上下文

目标：

- 不再隐式把发布中心筛选带到发布记录页。
- 从发布详情进入发布记录页时，当阶段内即可按 `release_request_id` 生效。

建议步骤：

1. 发布中心“发布记录”按钮改为进入 `/deploys`，并重置发布记录筛选为默认全部。
2. 发布详情“关联发布记录”进入 `/deploys?release_request_id=...`。
3. 在本阶段就解析 `/deploys?release_request_id=...` 并写入 `deployListFilters.releaseRequestID`；不要等到完整 URL 同步阶段。
4. 本次不新增“按当前筛选查看记录”入口。

验收：

- 从发布中心进入发布记录页，看到全部记录。
- 从发布详情进入发布记录页，只看到当前发布单的记录。
- 页面上能看懂当前筛选来源。

### 阶段 3：URL query 同步

目标：

- 筛选状态刷新不丢失，链接可复制。

建议步骤：

1. 扩展 `routeFromLocation()` 读取 `window.location.search`。
2. 扩展 `pathForPage()` 或新增 query 构造函数。
3. 发布中心筛选变化时同步到 `/releases?...`。
4. 发布记录筛选变化时同步到 `/deploys?...`。
5. 浏览器 `popstate` 时恢复筛选状态。
6. URL query 使用 snake_case：`service_id`、`environment_id`、`release_request_id`、`time_range`。

验收：

- 刷新页面后筛选状态保持。
- 浏览器返回/前进能恢复筛选。
- 复制 URL 给他人后能打开相同列表上下文。

### 阶段 4：提示条与清除动作

目标：

- 用户清楚知道为什么列表被过滤。

建议步骤：

1. 发布中心增加“清除筛选”动作。
2. 发布记录页增加筛选提示条。
3. 发布记录页提示条提供“查看全部记录”。

验收：

- 任意带上下文的发布记录页都能一键回到全部记录。
- 没有筛选时不显示多余提示。

### 阶段 5：服务端筛选与分页

目标：

- 数据量增长后避免全量加载。

建议步骤：

1. 新增 release list filter 输入结构。
2. 新增 deploy record list filter 输入结构。
3. repository 层使用参数化 SQL 拼接 where 条件。
4. handler 解析 query 参数。
5. 补 API 测试。

验收：

- 空 query 返回默认第一页。
- 空筛选与全量语义一致。
- 服务/环境/状态/release id 筛选结果正确。
- API key scope 行为保持不变。

## 10. 回归检查清单

前端行为：

- `/releases` 默认展示全部发布单。
- 清空服务筛选后刷新页面，仍是全部服务。
- 清空环境筛选后刷新页面，仍是全部环境。
- `/deploys` 默认展示全部发布记录。
- `/deploys?release_request_id=rel_xxx` 只展示该发布单记录。
- 发布详情返回发布中心时，不污染发布中心筛选。
- 工作台仍展示当前登录用户的待处理、运行中和最近失败发布。
- 创建发布单页面仍能正常选择服务、环境、版本和部署目标。

后端行为：

- `GET /api/v1/release-requests` 现有调用兼容。
- `GET /api/v1/deploy-records` 现有调用兼容。
- API key scope 不变。

验证命令：

```sh
make verify
```

涉及后端筛选或 repository 改动时，还需要执行：

```sh
make compose-check
```

如只完成前端状态拆分且未触及后端、migration、Worker、Compose 配置，可先执行前端 lint/build 所在的代码级检查；最终交付前仍建议跑 `make verify`。

## 11. Agent 执行注意事项

- 不要把 `selection` 简单改名后继续复用；必须拆出列表筛选和创建选择。
- 不要漏掉工作台；工作台需要独立偏好上下文，不能继续依赖发布中心或发布记录筛选。
- 不要让列表筛选进入 `keepOrFirst()` 或任何等价 fallback。
- 不要保留 `scoped` 作为发布记录筛选来源。
