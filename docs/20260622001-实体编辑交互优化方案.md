# 实体编辑交互优化方案

方案日期：2026-06-22

## 1. 背景

人工测试中发现，配置页（项目 / 服务 / 版本 / 环境 / 服务器 / 服务器组 / 部署目标）与系统页（用户 / 访问密钥 / 凭据 / 通知）的实体编辑交互存在不一致与缺口：

- 已创建的实体在清单列表中不可点击，必须滚动到页面底部的"维护栏"，在下拉里重新选中同一实体才能编辑。
- 部分实体创建后既无法编辑也无法删除（最严重的是凭据）。
- 配置页与系统页采用两套不同的交互范式，心智模型割裂。

本文给出从 P0 到 P3 的完整优化方案。**仅输出方案，不修改代码。**

## 2. 目标

- 统一实体"查看—编辑—停用—删除"的交互范式，消除滚动与重复选择。
- 补齐凭据等缺失的生命周期操作，消除安全卫生缺口。
- 明确"可删除 / 仅可禁用"的边界，避免破坏发布审计链。
- 不改变领域模型与表结构，不引入不可变实体的篡改风险。

## 3. 非目标

- 不改变版本的不可变性（版本不可编辑、不可删除）。
- 不对已被部署记录引用的运行实体（环境 / 服务器 / 服务器组 / 部署目标 / 项目 / 服务）提供硬删除。
- 不重构整个前端框架，不引入新的状态管理库。
- 不设计复杂 RBAC 或行级权限。

## 4. 现状事实矩阵

以下为代码核对后的真实能力，非主观判断。

### 4.1 配置页（三栏范式：清单只读 + 创建 + 维护栏）

| 实体 | 创建 | 编辑(PATCH) | 删除 | 启用/禁用 | 清单可点 | 独立编辑器 |
|------|:----:|:-----------:|:----:|:--------:|:--------:|:----------:|
| 项目 | 有 | 后端有，前端无 | 无 | 后端有，前端无 | 否 | 无 |
| 服务 | 有 | 后端有，前端无 | 无 | 后端有，前端无 | 否 | 无 |
| 版本 | 有 | 无 | 无 | 无 | 否 | 无 |
| 环境 | 有 | 有 | 无 | 有（维护栏） | 否 | 有 |
| 服务器 | 有 | 有 | 无 | 有（维护栏） | 否 | 有 |
| 服务器组 | 有 | 有 | 无 | 有（维护栏） | 否 | 有 |
| 部署目标 | 有 | 有 | 无 | 有（维护栏） | 否 | 有 |

### 4.2 系统页（列表 + 行内操作范式）

| 实体 | 创建 | 编辑 | 删除 | 启用/禁用 | 行内操作 |
|------|:----:|:----:|:----:|:--------:|:--------:|
| 用户 | 有 | 有 | 无 | 有内联 | 有 |
| 访问密钥 | 有 | 有 | 有内联 | 有内联 | 有 |
| 凭据 | 有 | 无 | 无 | 无 | 无（只读） |
| 通知 | 有 | 有 | 无 | 有内联 + 测试 | 有 |

### 4.3 关键缺口

- **凭据（Credential）**：只能创建，不能编辑、不能删除、不能禁用。凭据泄漏或需要轮换时 UI 无路可走。`internal/repository/credential.go` 仅有 `CreateCredential` / `ListCredentials` / `GetCredential` / `GetCredentialSecret`，无 Update / Delete。
- **项目 / 服务**：后端有 PATCH 接口，前端无编辑器，接口实际为死接口。
- **版本**：无 PATCH。这是正确设计（不可变制品），保留。
- **除访问密钥外无任何实体支持删除**；建错的环境 / 服务器 / 部署目标永久存在，只能禁用。
- `credentials` 表无外键约束，`servers.credential_ref` 为软引用字符串，删除凭据需先校验未被服务器引用。

## 5. 方案

### 5.1 设计原则（贯穿 P1–P3）

1. **在哪看在哪改**：清单行可点击，点击即进入编辑态，不再滚动到独立维护栏。
2. **范式统一**：配置页收敛为系统页已有的"可点列表 + 内联操作 + 创建"范式。
3. **删除边界显式化**：被引用实体一律"禁用不删除"；未被引用的凭据 / 通知 / 访问密钥允许删除；UI 上只对可删除实体露出删除按钮。
4. **不可变实体保持不可变**：版本不提供编辑 / 删除入口。

### 5.2 P0：凭据生命周期补齐（安全缺口）

**问题**：凭据无法禁用、删除、轮换，泄漏后无法在 UI 内处置。

**后端**：

- 新增 `PATCH /api/v1/credentials/{id}`，支持 `name`、`description`、`enabled` 字段；不允许通过 PATCH 改 `secret`（轮换走专门语义，见下）。
- 新增 `DELETE /api/v1/credentials/{id}`，删除前校验：若存在**任何** `servers.credential_ref = {id}` 的服务器引用（含已禁用服务器），返回 `409 conflict`，提示"凭据仍被服务器引用，请先解除引用"。校验在应用层完成（无外键约束）。
- `DeleteCredential` 的引用检查与删除必须在**同一事务**内完成，避免应用层先查后删的 TOCTOU 并发窗口；事务内 `SELECT count(*) FROM servers WHERE credential_ref = ?`，命中即回滚并返回引用冲突错误。
- `internal/repository/credential.go` 新增 `UpdateCredential`、`DeleteCredential`（事务版）。
- `internal/app/credential.go` 的 `CredentialService` 暴露 `Update` / `Delete` 方法，`Delete` 返回引用冲突错误供 HTTP 层稳定映射为 `409 conflict`。
- **关联增强（本次一并实现）**：服务器创建 / 更新时校验 `credential_ref` 引用完整性。`createServer` / `patchServer` 在 `auth_type != "none"` 且 `credential_ref` 非空时，校验该凭据存在且 `enabled = true`，否则返回 `400 invalid_argument`。这使"禁用凭据"不会在运行时才暴露失败，与 P0 的禁用能力形成闭环。
- 凭据轮换（替换 secret）第一版**不单独做端点**；通过"删除旧凭据 + 创建新凭据 + 服务器改引用"完成，KISS。如后续需要，再补 `POST /credentials/{id}/rotate`。

**前端**：

- 凭据列表行从只读 `EntityList` 改为可点列表，行内暴露"编辑""禁用/启用""删除"。
- 凭据编辑抽屉/表单：`name`、`description`、`enabled` 可改；`type`、`secret` 创建后不可改（提示"如需更换密钥请删除后重建"）。
- 禁用被服务器引用的凭据时，UI 需列出受影响的服务器（前端用 `state.servers` 过滤 `credential_ref === id` 计算），提示用户先解除引用或改用其他凭据。
- 删除按钮带 `Popconfirm`，引用冲突时展示后端返回的提示。

**涉及文件**：
- `internal/httpapi/router.go`、`internal/httpapi/credential.go`
- `internal/httpapi/inventory.go`（`createServer` / `patchServer` 增凭据引用校验）
- `internal/app/credential.go`
- `internal/repository/credential.go`
- `web/src/App.tsx`（新增 `CredentialEditor` / `CredentialList` 可点列表）

**风险**：
- 删除凭据的引用校验统一覆盖"启用 + 禁用"的全部服务器，任何引用都阻断删除，避免启用服务器时才发现凭据已丢。
- 凭据 `secret` 不允许 PATCH 是为了防止静默更换破坏审计；轮换走删除重建路径。
- 服务器创建/更新新增凭据校验后，已有"引用已禁用凭据"的存量服务器在下次更新时会被拒绝；属预期行为，提示用户先启用或更换凭据。

### 5.3 P1：配置页交互范式统一（核心别扭点）

**问题**：清单不可点击，编辑需滚动到底部维护栏重新选择同一实体；与系统页范式不一致。

**方案**：将配置页"清单(只读) + 创建 + 维护栏"三栏，收敛为"可点列表 + 内联操作 + 创建"。

**交互**：

- 清单行可点击，点击后在行内展开或弹出抽屉，直接进入编辑表单，回填该实体字段。
- 编辑表单复用现有 `EnvironmentEditor` / `ServerEditor` / `ServerGroupEditor` / `DeploymentTargetEditor` 的字段逻辑，但从"下拉选择 + 表单"改为"点击行即载入"。
- 行内操作：
  - 环境：启用/禁用开关、冻结开关（独立）、编辑。
  - 服务器：启用/禁用、编辑、测试连接（保留已实现的 `POST /servers/{id}/test`）。
  - 服务器组：启用/禁用、编辑。
  - 部署目标：启用/禁用、编辑。
- 删除底部独立的"维护与发布保护""维护部署目标"区块（`infrastructure-maintenance`），其能力上移到可点列表行内。
- 保留 `INVENTORY`（清单）与 `CREATE`（创建）两栏，但清单从只读升级为可交互。

**涉及文件**：
- `web/src/App.tsx`：`EntityList` 抽象出可点击变体或新增 `EditableEntityList`；改造 `884`/`889`/`894`/`895` 的配置页布局；移除 `890`/`895` 维护栏区块。
- 不涉及后端（PATCH 接口已存在）。

**风险**：
- 行内展开 vs 抽屉的选择影响实现量。建议**抽屉（Drawer）**，与编辑表单字段较多（部署目标有 8 个字段）的诉求匹配，且不挤压列表空间。需引入 antd `Drawer`（已在 antd 依赖内，无需新增）。
- 改动集中在配置页布局，需回归环境冻结、服务器测试连接等已有交互不回归。
- 配置页单个 JS chunk 已超 500kB（见 `development-completion-audit.md`），本次改动会进一步增大；属非阻断项，但应在后续单独做代码分割。
- **并发策略**：MVP 采用 **last-write-wins**，现有所有更新接口无版本号 / `updated_at` 条件 / 冲突检测，本次不引入乐观锁。影响：两个管理员同时编辑同一实体时，后保存者覆盖前者，无提示。记录此为已知取舍，后续如需冲突保护，最小方案是提交时带 `updated_at`，后端不匹配返回 `409`，前端提示刷新后重试——作为后续增强项，不在本次实现。

### 5.4 P2：项目 / 服务前端编辑器补齐（死接口激活）

**问题**：项目 / 服务后端有 PATCH，前端无编辑器，接口闲置；用户创建后无法改名 / 改描述 / 禁用废弃项目。

**方案**：

- 随 P1 的可点列表一起，为项目、服务补行内编辑入口。
- 项目编辑表单字段：`name`、`slug`、`description`、`enabled`（与后端 `patchProject` 字段一致）。
- 服务编辑表单字段：`name`、`slug`、`description`、`enabled`（与后端 `patchService` 字段一致）。**不支持改 `project_id`**——后端 `patchService` / `UpdateService` 均不接收该字段，服务归属项目在创建时确定，不支持迁移；如确需迁移，删除旧服务在新项目下重建。
- 新增 `ProjectEditor`、`ServiceEditor` 组件，字段逻辑参照对应 `Form`。

**涉及文件**：
- `web/src/App.tsx`：新增 `ProjectEditor`、`ServiceEditor`；改造 `884` 配置页"应用与版本"区块的清单为可点列表。
- 不涉及后端。

**风险**：
- 改 `slug` 可能影响以 slug 为 key 的外部引用（如 CI 调用）。建议 UI 上对 `slug` 标注"修改可能影响已配置的 CI 引用，请谨慎"，但不强制禁止。
- 服务 `project_id` 不支持编辑是后端既定边界，UI 上不展示该编辑项，避免出现"改了不生效"的死字段。
- 改 `project_id` 会移动服务归属，需确保部署目标 / 版本的级联展示正确刷新（现有 `refreshAll` 已覆盖）。

### 5.5 P3：删除策略显式化

**问题**：当前"只有访问密钥能删"是隐含规则，用户找不到删除按钮会误判为 bug。

**方案**：明确并文档化删除边界，UI 按边界一致露出操作。

**删除边界（最终规则）**：

| 实体 | 删除 | 原因 |
|------|:----:|------|
| 项目 | 否（仅禁用） | 可能被服务 / 部署记录引用 |
| 服务 | 否（仅禁用） | 可能被版本 / 部署目标引用 |
| 版本 | 否 | 不可变制品，引用即审计 |
| 环境 | 否（仅禁用 + 冻结） | 被部署记录引用 |
| 服务器 | 否（仅禁用） | 被部署记录 / 服务器组引用 |
| 服务器组 | 否（仅禁用） | 被部署目标引用 |
| 部署目标 | 否（仅禁用） | 被发布单引用 |
| 用户 | 否（仅禁用） | 被发布单 / 事件审计引用 |
| 凭据 | 是（引用校验后） | P0 新增；未被服务器引用时可删 |
| 通知配置 | 是 | 不影响发布主流程，可清理 |
| 访问密钥 | 是（已有） | 归属用户可删自己的 |

**后端**：

- 通知配置新增 `DELETE /api/v1/notification-configs/{id}`（`internal/repository/notification.go` 已有 `UpdateNotificationConfig`，补 `DeleteNotificationConfig`）。
- 其余被引用实体**不新增 DELETE 接口**，UI 上只露"禁用"。

**前端**：

- 可删除实体（凭据 / 通知 / 访问密钥）行内露"删除"按钮 + `Popconfirm`。
- 仅可禁用实体行内只露"禁用 / 启用"开关 + "编辑"，不露删除。
- 在配置页与系统页顶部或帮助文案中，用一句话说明"已被引用的对象只能禁用，不能删除"，消除"找不到删除"的困惑。

**通知删除的审计展示规则（硬删除，明确实现）**：

- 坚持硬删除，不改表结构、不引入 `deleted_at` 软删除（违背"不改变领域模型与表结构"非目标）。
- 通知配置已有 `enabled` 字段且 UI 已支持禁用，"不想用就禁用"的路径已存在；硬删除的价值是**彻底清理建错 / 重复的配置**，与禁用语义不同，不可互相替代。
- 删除后历史 `notification_deliveries.config_id` 成为悬空引用，**不做级联删除**（保留投递审计）。前端投递记录列表在 `config_id` 找不到对应配置时显示"已删除配置"——此展示非现有链路自然结果，需在前端 `EntityList` 渲染投递记录时加一层 `findByID(state.notificationConfigs, config_id)` 回退判断，故纳入本次前端实现范围。
- 后续若需更强审计（如保留删除前的配置名称快照），作为独立后续项，不在本次实现。

**涉及文件**：
- `internal/httpapi/router.go`、`internal/httpapi/notification.go`、`internal/repository/notification.go`
- `web/src/App.tsx`（行内按钮一致性 + 文案 + 投递记录"已删除配置"回退展示）

**风险**：
- 通知配置硬删除后，排查"哪条配置曾经发送失败"会略受影响；通过保留投递记录 + "已删除配置"展示缓解，不阻断。
- 删除边界需与 `docs/api-design.md` 同步记录，避免后端接口与文档不一致。

## 7. 潜在风险与回滚

- **审计完整性**：所有"仅禁用"决策均出于保护发布审计链，硬删除只限未被引用的辅助实体。若误删凭据导致服务器连接失败，靠引用校验前置拦截；若仍发生，需从数据库恢复。
- **前端体积**：配置页改造增大 chunk，属非阻断性能项，后续单独代码分割。
- **交互回归**：P1 改动面较大，需回归环境冻结、服务器测试连接、部署目标编辑、服务器组成员维护等已有路径。
- **回滚**：后端新增接口为增量，回滚即移除路由与 handler；前端改动集中在配置页与凭据/通知列表，可按 P0–P3 分批合入，互不阻塞。

### 7.1 关联后续项（本次不实现，仅记录）

以下为评审发现的真实缺口，但超出"实体编辑交互优化"主题或需动表结构，作为独立后续项记录，不在本次实现：

- **preflight 凭据可用性检查**：当前 preflight（`internal/app/release.go`）不检查凭据，SSH 执行时才取凭据失败。P0 补了服务器创建/更新的凭据引用校验后，存量"引用已禁用凭据"的服务器仍可能在 preflight 通过后运行失败。后续应在 preflight 增加凭据可用性检查，作为发布链路增强项单独实现，避免本方案改动扩散到发布核心逻辑。
- **实体变更审计**：当前系统所有实体变更均不写 `ReleaseEvent`（只有发布动作写事件流）。凭据 / 通知删除补审计需单独建审计表或扩展事件模型，超出本次范围，作为独立后续项。

## 8. 建议实施顺序

1. **P0 凭据生命周期**：安全缺口优先，后端 + 前端独立可交付。
2. **P1 配置页范式统一**：核心体验问题，纯前端，收益最大。
3. **P2 项目/服务编辑器**：随 P1 顺带完成，边际成本低。
4. **P3 删除策略显式化**：含通知 DELETE 后端，收尾一致性。

P0 与 P1 可并行（前者后端为主，后者前端为主）；P2 依赖 P1 的可点列表组件；P3 可在 P0/P1 完成后统一收尾。

## 9. 验证计划

**后端**：
- 新增 handler 的单元测试：凭据引用冲突返回 `409`、凭据删除事务、凭据更新、通知删除、服务器创建/更新引用不存在或已禁用凭据返回 `400`。

**前端**：
- `npm run lint` + `npm run build`（`web/package.json` 仅有 `lint/build/dev`，无前端测试脚本，交互回归靠手工）。

**前端最小手工回归清单**（P1 改动面大，必须逐项验证）：
1. 项目编辑：点击行 → 抽屉回填 → 改名/描述/启用 → 保存刷新生效。
2. 服务编辑：点击行 → 抽屉回填 → 改名/slug/启用 → 保存刷新生效（确认无 `project_id` 编辑项）。
3. 环境冻结：启用/禁用开关、冻结开关独立切换，冻结后新建发布被阻断。
4. 服务器测试连接：编辑抽屉内测试连接按钮，返回成功/失败提示。
5. 服务器组成员编辑：点击行 → 抽屉回填成员 → 增删成员保存生效。
6. 部署目标编辑：点击行 → 抽屉回填 8 字段 → 修改执行器/脚本保存生效。
7. 凭据禁用/删除：禁用被引用凭据时列出受影响服务器；删除被引用凭据返回 409 提示；删除未引用凭据成功。
8. 通知删除：删除通知配置后，投递记录列表对应 `config_id` 显示"已删除配置"。

**集成**：
- `make compose-check` 验证凭据删除引用校验、服务器引用凭据校验、配置页编辑后发布流程不回归。
