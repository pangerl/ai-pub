# 用户管理与 demo 账号保护整改方案

## 1. 背景

当前管理控制台的用户管理存在三个问题：

1. 管理员查看“现有用户”时可能看到旧数据，新创建用户没有稳定出现在列表中。
2. 前端用户列表只有启用/禁用操作，没有编辑入口。
3. 如果公网 demo 账号拥有管理员权限，访客可能禁用、降级或改坏入口账号，导致后续访客无法登录；同时也需要避免访客修改内置 `admin` 账号。

本方案目标是补齐 MVP 需要的用户维护能力，同时保留简单边界，不引入复杂 RBAC、组织、多租户或审计账本。

## 2. 当前实现依据

- `GET /api/v1/users` 已存在。管理员会返回完整用户字段，普通用户只返回用户目录。
- `POST /api/v1/users` 已存在，并允许管理员创建用户和设置初始密码。
- `PATCH /api/v1/users/{id}` 已存在，当前支持更新 `display_name`、`role`、`enabled`。
- `DELETE /api/v1/users/{id}` 支持管理员会话删除用户；删除时同步删除该用户归属的 API Key，历史发布记录保留原用户 ID。
- repository 层已有 `SetUserPassword`，但没有 HTTP handler。
- 当前前端 `UserList` 只提供启用/禁用按钮，没有用户编辑抽屉。
- 当前 demo 加固文档仍把“无密码修改接口”列为可接受残余风险；本次改造后需要同步更新该边界。

## 3. 设计原则

1. 后端是安全边界。前端隐藏按钮只能改善体验，不能作为保护措施。
2. `admin` 是系统保留管理员账号，不能被其他管理员破坏。
3. demo 入口账号必须可持续登录，不能被访客改坏。
4. 用户可删除，但删除只清理登录身份和访问密钥；历史发布单、事件记录等保留原用户 ID，不因历史记录阻止删除。
5. 密码重置不放进普通编辑表单，单独作为高风险动作处理。
6. 第一版只处理 `admin` / `demo` 这类保留账号，不引入通用策略引擎。

## 4. 目标规则

### 4.1 生产通用规则

系统默认将 `admin` 视为保留账号：

- 不允许删除 `admin`、demo 保护账号、当前登录用户自己，以及最后一个已启用管理员。
- 不允许修改用户名；继续不在 PATCH 中支持 `username`。
- 非 `admin` 本人不允许修改 `admin` 账号信息。
- 非 `admin` 本人不允许重置 `admin` 密码。
- `admin` 不允许被禁用。
- `admin` 不允许被降级为 `employee`。
- 不允许禁用最后一个已启用管理员。
- 不允许将最后一个已启用管理员降级为普通员工。

说明：

- “`admin` 本人”指当前登录会话用户的 `username == "admin"`。
- API Key 即使拥有 `admin:write`，也不允许修改 `admin` 账号或重置 `admin` 密码。原因是 API Key 没有人类登录会话确认，不适合接管根管理员账号。

### 4.2 demo 模式规则

公网 demo 场景额外保护入口账号 `demo`：

- `demo` 不允许被禁用。
- `demo` 不允许被降级。
- `demo` 不允许被重置密码。
- `demo` 的显示名可选：建议也禁止修改，避免访客把入口账号改成误导性名称。
- demo 管理员可以创建、编辑、禁用其他试用用户，但不能修改 `admin` 和 `demo`。

推荐新增配置：

```env
DEMO_MODE=false
DEMO_PROTECTED_USERNAMES=demo
```

有效保护集合：

- 非 demo 模式：`admin`
- demo 模式：`admin` + `DEMO_PROTECTED_USERNAMES`

`DEMO_PROTECTED_USERNAMES` 默认为 `demo`，以便 `DEMO_MODE=true` 时开箱保护公网入口账号。

## 5. 后端改造

### 5.1 配置

在 `internal/config.Config` 中新增：

- `DemoMode bool`
- `DemoProtectedUsernames string`

加载规则：

- `DEMO_MODE` 使用现有 `envBool`。
- `DEMO_PROTECTED_USERNAMES` 默认 `demo`。
- 解析时 trim 空格，忽略空项。

建议封装最小 helper：

- `protectedUsernames(cfg config.Config) map[string]bool`
- `isReservedAdmin(user domain.User) bool`
- `isDemoProtected(user domain.User, cfg config.Config) bool`
- `isProtectedUser(user domain.User, cfg config.Config) bool`

### 5.2 PATCH 用户接口

将 `patchUser` 从只依赖 `store` 调整为可读取配置和当前操作者：

```go
func patchUser(store repository.Store, cfg config.Config) http.HandlerFunc
```

校验顺序：

1. 读取目标用户。
2. 读取当前会话用户。若请求来自 API Key，则仅允许普通非保护用户维护；禁止触碰保护用户。
3. 应用 patch 前先判断目标用户是否保护账号。
4. 对保护账号执行强制规则。
5. 对所有用户执行“最后一个 enabled admin”保护。
6. 调用 `store.UpdateUser`。

错误建议：

- 修改保护账号返回 `403 forbidden`，错误码可用 `protected_user`。
- 禁用或降级最后管理员返回 `400 invalid_argument`，错误码可用 `last_admin_required`。

### 5.3 最后管理员保护

repository 增加一个查询方法：

```go
CountEnabledAdmins(ctx context.Context) (int, error)
```

在更新用户时，如果目标用户当前是 `role=admin && enabled=true`，并且 patch 会导致其不再是 enabled admin，则查询当前 enabled admin 数量。数量小于等于 1 时拒绝。

这个检查应在 HTTP 层完成即可，MVP 不必先做数据库触发器。

### 5.4 重置密码接口

新增接口：

```http
POST /api/v1/users/{id}/password
```

请求体：

```json
{
  "password": "new-password"
}
```

规则：

- 仅管理员会话可调用。不要允许 API Key 调用。
- 目标是普通用户：任意管理员可重置。
- 目标是 `admin`：只有当前登录的 `admin` 本人可重置。
- 目标是 demo 保护账号：demo 模式下禁止重置。
- 密码沿用 `auth.HashPassword`，至少 8 位。
- 成功后调用 `SetUserPassword`，使 `session_version + 1`，旧会话失效。

不做：

- 不做“用户自行修改密码”页面。
- 不做旧密码校验。
- 不做密码复杂度策略。

这些可以后续在个人设置页补。

### 5.5 创建用户接口

`POST /api/v1/users` 保持现状，但建议补充：

- trim `username` 和 `display_name`。
- 禁止创建重复用户名由数据库唯一约束兜底。
- 不允许创建用户名为空的用户；如果当前 schema 未兜住，HTTP 层补校验。

不要允许通过创建接口覆盖或重置 `admin` / `demo`。

## 6. 前端改造

### 6.1 用户列表刷新

新增局部刷新函数：

```ts
async function refreshUsers() {
  const users = await apiGet<Entity[]>('/api/v1/users');
  setState((current) => ({ ...current, users }));
}
```

使用位置：

- 新建用户成功后调用 `refreshUsers()`。
- 编辑用户成功后调用 `refreshUsers()`。
- 启用/禁用成功后调用 `refreshUsers()`。
- 重置密码成功后不必全量刷新，但可以刷新用户列表以同步更新时间。

`refreshAll()` 仍用于页面初始化和跨模块刷新，不再作为用户管理动作的唯一刷新路径。

### 6.2 用户编辑抽屉

新增 `UserEditDrawer` 或让 `UserList` 接收 `onOpen`：

可编辑字段：

- 显示名
- 角色
- 启用状态

不可编辑字段：

- 用户名
- 创建时间
- 密码

交互：

- 行点击打开编辑抽屉。
- 行内保留启用/禁用快捷按钮。
- 对保护账号禁用危险操作按钮，并展示简短说明，例如“保留账号不可禁用”。
- 前端保护只作为提示；后端仍必须拒绝非法请求。

### 6.3 重置密码入口

在用户编辑抽屉中增加独立按钮“重置密码”：

- 点击后打开二次确认或小弹窗输入新密码。
- 对 `admin`：
  - 当前登录用户不是 `admin` 时隐藏或禁用。
- 对 demo 保护账号：
  - demo 模式下隐藏或禁用。
- 成功后提示“密码已重置，旧会话已失效”。

为了避免把 demo 配置暴露成复杂前端状态，第一版可以不新增 `/config` 接口。前端按当前用户和用户名做基础禁用提示，最终以后端错误为准。

`deploy/compose.demo.yaml` 增加：

```yaml
DEMO_MODE: "true"
DEMO_PROTECTED_USERNAMES: ${DEMO_PROTECTED_USERNAMES:-demo}
```

### 7.2 文档同步

更新：

- `docs/demo-public-hardening.md`
- `deploy/README.md`
- `README.md` 如有必要补充“demo 入口账号受保护”

将“无密码修改接口”从残余风险中移除或改写为：

- demo 模式下保护账号禁止通过页面/API 修改密码。
- 普通试用用户仍可能被访客修改，demo 数据会定期重置。

## 8. 测试计划

### 8.1 后端测试

新增或扩展 `internal/httpapi/session_test.go` / `inventory_test.go`：

1. 管理员可修改普通用户显示名、角色、启用状态。
2. 非 `admin` 本人的管理员不能禁用 `admin`。
3. 非 `admin` 本人的管理员不能降级 `admin`。
4. 非 `admin` 本人的管理员不能重置 `admin` 密码。
5. `admin` 本人可以重置 `admin` 密码。
6. 不能禁用最后一个 enabled admin。
7. 不能降级最后一个 enabled admin。
8. `DEMO_MODE=true` 时不能禁用、降级或重置 `demo`。
9. API Key 即使具备 `admin:write` 也不能修改 `admin` 或重置密码。

专项命令：

```bash
go test ./internal/httpapi ./internal/repository
```

### 8.2 前端检查

至少运行：

```bash
npm run lint
npm run build
```

手工检查：

1. 管理员进入“系统 / 用户与权限”。
2. 新建用户后，列表不刷新浏览器也能出现新用户。
3. 编辑普通用户显示名、角色、状态后，列表立即同步。
4. 重置普通用户密码后，旧密码不能登录，新密码可登录。
5. `admin` 和 demo 保护账号的危险操作不可用，直接调 API 也会失败。

### 8.3 demo 验收

涉及 demo 配置时运行：

```bash
make demo-check
```

如改到 Compose demo 启动链路，补充手工验证：

1. demo 账号可登录。
2. demo 账号拥有管理页面体验权限。
3. demo 账号不能修改 `admin`。
4. demo 账号不能禁用、降级、重置 `demo` 自己。
5. 普通试用用户仍可被创建和编辑。

## 10. 不做范围

- 不做用户软删除、归档状态或历史发布记录级联清理。
- 不做复杂 RBAC、权限组、组织、多租户。
- 不做用户自助修改密码。
- 不做 OAuth、SSO、MFA。
- 不做不可篡改审计账本。
- 不做数据库触发器级账号保护。

## 11. 完成标准

本整改完成后应满足：

1. 管理员能在管理控制台看到完整用户列表，新建后列表立即更新。
2. 管理员能编辑普通用户显示名、角色、启用状态。
3. 管理员能重置普通用户密码。
4. `admin` 账号不能被其他管理员、demo 管理员或 API Key 禁用、降级、改资料或重置密码。
5. 系统不会出现 0 个 enabled admin。
6. demo 模式下 `demo` 账号不能被禁用、降级或重置密码。
7. demo 用户仍可体验管理员视角，但不能破坏 demo 入口和根管理员账号。
