# 发布系统产品需求文档（PRD）

## 1. 文档目标

本需求文档回答三个问题：

1. 这个项目本质上要解决什么问题。
2. 第一版产品必须具备哪些能力和业务约束。
3. 哪些需求属于明确非目标，避免新项目一开始过度设计。

本文不是接口详细设计，也不是开发排期。后续可基于本文再拆分 Go 技术方案、领域模型设计、API 设计、前端信息架构、工程结构和开发任务清单。

## 2. 产品定位

系统定位为：

> 面向研发和运维、支持 AI 调用、现代化、轻量可审计的服务发布执行系统。

它不是完整 DevOps 平台，也不是审批流或 SOP 系统。项目的核心价值是把服务版本选择、发布前检查、执行门禁、执行编排、服务器日志、失败恢复和审计证据串成一个稳定的执行闭环。

新系统应优先服务以下目标：

- 让研发、运维和 AI Agent 可以方便、低成本、高效、安全地完成版本发布。
- 让研发和运维可以用统一执行入口完成服务发布。
- 让生产发布具备管理员确认门禁、回滚辅助、审计和风险控制。
- 让非生产发布保持足够自助和高效。
- 让 CI/CD 和 AI Agent 作为受限调用方接入，而不是绕过执行边界。
- 让执行方式可替换，SSH 只是内置执行器之一。
- 让系统可以被开源用户低成本部署、体验和二次开发。

第一版即使不接入 AI Agent，也必须有独立价值：用比完整 DevOps 平台更轻的成本，把“发布什么版本、发到哪个环境、由谁执行、执行到哪些服务器、是否成功、失败如何追溯、如何重新发布或回滚”沉淀成统一执行证据链。AI Agent 是后续放大入口效率的增强能力，不是第一版成立的前提。

第一版目标用户画像和规模假设：

- 适合 5-50 人研发团队或单一组织内部工具团队。
- 典型规模为 5-30 个服务、3-5 套环境、10-100 台服务器。
- 典型发布频率为每天 1-30 次发布，生产发布占比较低但要求执行可追踪。
- 第一版优先解决单组织、功能闭环和发布执行稳定的问题，不追求大型平台级多租户、复杂组织流转或高并发性能场景。

与常见方案的边界差异：

- 相比 Jenkins/GitLab CI：本系统不替代构建流水线，重点补齐发版执行入口、preflight、执行队列、服务器日志、失败恢复和审计证据。
- 相比 ArgoCD/Spinnaker：本系统第一版不做复杂编排、环境晋升和声明式 GitOps，同样发布动作可以通过后续执行器接入这些工具。
- 相比自研 SSH 脚本：本系统不只是执行脚本，而是给脚本执行增加统一状态机、执行门禁、队列、日志和追责边界。

第一版成功度量：

- 北极星指标：通过统一发布执行入口完成的发布占比持续提升。
- 效率指标：非生产发布从创建到执行开始的中位时长、生产发布确认等待中位时长。
- 质量指标：发布成功率、失败后可定位到服务器级错误的比例、回滚发布单使用比例。
- 护栏指标：生产发布绕过发布执行记录次数为 0，生产发布绕过已配置门禁次数为 0，审计事件缺失的关键动作次数为 0。
- 第一版上线第一个月作为指标基线期；基线期后再设定下一阶段目标值，避免在缺少历史数据时虚设数字。

## 3. 背景与问题

许多团队的发布动作长期散落在 CI、聊天工具、人工脚本和口头确认之间。外部 OA、SOP 或审批流不一定存在，也不一定应该由发布系统承载；但执行层仍需要清楚知道发布什么、发到哪里、谁执行、是否可执行、执行到哪些服务器、失败后如何追溯。

AI Agent 参与研发和运维流程后，这个问题会更明显：AI 可以帮助选择服务版本、发起非生产发布、查询状态和总结结果，但如果系统没有统一的执行入口、preflight、权限和审计边界，AI 很容易变成新的旁路入口。

因此，新项目的核心不是“再做一个审批工单系统”，而是做一个围绕服务发布执行的底层运维工具：

- 发布执行有统一入口。
- 发布前有可解释检查。
- 生产发布必须由管理员确认，非生产发布由执行人本人确认。
- 自动化和 AI 只能在权限和策略内操作。
- 执行过程有状态、日志和审计证据。
- 不同团队可以接入自己的执行方式。

## 4. 产品原则

### 4.1 统一执行入口

所有真实发布动作必须进入发布系统的执行流程。

Web、API、CI/CD 和 AI Agent 可以有不同交互入口，但它们最终都必须进入同一套发布执行单、preflight、执行策略、队列、日志和审计流程。

### 4.2 发布单与执行分层

`ReleaseRequest` 表达一次发布执行意图、轻量门禁状态、来源上下文和审计事件。

`DeployRecord` 表达真实执行记录、执行状态、服务器日志和执行器结果。

未通过执行门禁前不应创建真实执行任务。确认通过后，可以创建 `DeployRecord`，进入队列或执行状态。

确认通过时，应由应用服务在同一事务内完成发布单状态更新、`DeployRecord` 创建、执行快照生成和服务器日志初始化，确保 `ReleaseRequest.queued` 只有一个写入入口。

状态归属规则：

- `ReleaseRequest` 是执行意图和门禁状态的权威对象，负责提交、预检、确认、驳回、取消、回滚关联和最终结果摘要。
- `DeployRecord` 是执行过程的权威对象，负责排队、运行、服务器结果、执行器结果和部分成功。
- 发布单进入执行后，其最终成功/失败应由 `DeployRecord` 和服务器日志聚合后回写，避免两个对象各自独立判断终态。
- 页面展示可以聚合发布单和发布记录，但写入状态时必须有单一来源。

发布单不是传统意义上“重流程、重表单、层层流转”的工单系统。它是一次发布执行的最小上下文载体，用于让一次发布可选择、可检查、可确认、可执行、可追溯、可恢复。

### 4.3 AI 是受限调用方，不是旁路发布引擎

AI Agent 只能通过受限 API 创建发布单、执行预检、查询状态和总结结果。

AI 不能绕过发布执行流程直接操作执行器。生产发布必须由管理员确认，AI 不能独立完成生产确认。

### 4.4 发布保护简单可解释

第一优先级是可理解、可审计、可维护，而不是复杂流程引擎。

生产环境固定由管理员确认；非生产环境固定由发起人本人确认。环境可单独冻结发布，不提供服务级、系统级或合并策略。

### 4.5 执行器可替换，但执行流程稳定

系统应抽象执行器，让 SSH、Mock、Webhook、Jenkins、Kubernetes、ArgoCD 等执行方式可扩展。

但无论执行器如何变化，发布单、preflight、环境发布保护、确认、队列、日志和审计流程应保持统一。

第一版只内置：

- Mock/Dry-run Executor：用于本地体验、演示和安全验证。
- SSH Executor：用于最小生产可用发布。

Webhook、Jenkins、Kubernetes、ArgoCD 等执行器作为后续扩展，不进入第一版内置范围。

### 4.6 面向 Go 语言重新实现

后续技术实现计划使用 Go。本文不规定具体框架、数据库、前端技术栈或目录结构，但产品需求应支持 Go 服务端清晰表达以下边界：

- 领域模型。
- 应用服务。
- API 层。
- 执行器插件或适配层。
- 后台调度任务。
- 审计事件。
- 配置和权限。

## 5. 角色与调用身份

第一版面向创业公司和小公司，默认没有复杂组织架构，不支持多租户，只设置两种人类角色，避免过早引入复杂 RBAC、项目审批人、发布经理、环境负责人等传统流程角色。

人类角色：

- 普通员工。
- 管理员。

调用身份不等于人类角色。系统还需要记录 AI Agent、CI/CD 和系统任务等非人类调用方，用于权限控制和审计追责。

调用身份：

- `user`：员工自己在 Web 或 API 操作。
- `ai_agent`：AI Agent 代表员工操作。
- `service_account`：CI/CD 或系统集成使用服务账号操作。
- `system`：系统自动动作，例如队列调度、状态修复、通知重试。

`ai_agent` 只用于表达调用来源和审计语义，不是新的人类角色，也不要求第一阶段建立独立 Agent 身份表、Agent 注册中心或完整 Agent RBAC。实际权限仍应落到被代表的用户会话、受限 API Key 或服务账号 scope 上。

### 5.1 普通员工

诉求：

- 查找自己需要发布的服务。
- 选择目标环境和版本。
- 第一版通过 Web/API 提交发布单，后续可通过 AI Agent 代理提交。
- 查看确认状态、发布进度和日志。
- 在环境未冻结时自助完成非生产发布。
- 在失败后发起回滚或重新发布。

权限边界：

- 可以提交发布单。
- 第一版不做项目级、服务级资源 ACL；普通员工默认可查看和提交系统中启用的项目、服务、环境和发布记录。
- 可以执行或确认环境发布保护允许的发布。
- 不能绕过环境发布保护直接操作执行器。
- 不能维护高风险系统配置。

### 5.2 管理员

诉求：

- 管理用户、项目、服务、环境、服务器、网关、部署目标和执行器。
- 查看系统健康、发布队列、失败率和执行器可用性。
- 处理需要管理员确认的生产发布。
- 排查发布失败、队列阻塞和僵尸任务。

权限边界：

- 可以维护高风险配置。
- 可以确认需要管理员确认的生产发布。
- 可以配置策略、冻结开关、API Key 和执行器。
- 不能绕过发布单、preflight、审计和执行记录直接发布。

### 5.3 CI/CD 与系统集成

诉求：

- 注册服务版本和构建信息。
- 可选地创建发布单。
- 查询发布结果。
- 通过 API Key 和 scope 进行最小权限访问。

第一版外部 CI 版本登记使用统一接口。CI 传递发布系统可读的 `project_key` 和 `service_key`，系统据此解析内部服务；CI 不需要保存或传递 `service_id`，也不为每个服务维护不同回调地址。GitLab CI 是首个接入示例。

权限边界：

- 只能执行被 scope 授权的操作。
- 不能因为 API Key 由管理员创建就自动拥有管理员权限。
- 不能绕过已配置的生产发布门禁。

### 5.4 后续 AI Agent 调用方

AI Agent 是后置增强能力，不进入第一版基础执行闭环验收。第一版只需要在调用身份、审计事件和 API 设计上预留 AI Agent 所需字段。

后续接入时：

- AI Agent 本身不是人类角色。
- AI Agent 必须代表某个授权员工或使用受限服务账号操作。
- AI Agent 不能绕过生产发布门禁；生产发布必须由管理员确认。
- AI Agent 的所有关键动作必须写入审计事件。
- Agent 相关元数据只保存最小审计上下文，例如 agent 名称、skill 版本、用户确认后的发布意图摘要、客户端请求 ID 或外部会话引用；不保存完整对话、prompt、模型推理过程或 token 明细。

## 6. 核心业务对象

核心关系：

```text
Project
  ├── Service
  │     └── ServiceVersion
  ├── Environment
  └── DeploymentTarget
        = Service + Environment + Executor + RuntimeTarget

Infrastructure
  ├── Server
  ├── Gateway
  ├── ServerGroup
  └── Executor
```

说明：

- `Project` 是业务系统边界。
- `Service` 是可发布单元，必须属于一个项目。
- `ServiceVersion` 是服务维度的历史构建版本，必须长期保留。
- `Environment` 是项目内的逻辑发布目标，必须属于一个项目。
- `Server` 是基础设施资源，不直接属于某个服务。
- `DeploymentTarget` 是连接服务、环境和实际运行目标的关键配置对象。
- 发布时用户不直接选择服务器，而是选择服务、环境和版本；系统根据部署目标解析实际执行方式。

### 6.1 项目 Project

项目是服务、环境和发布治理的组织边界。

需求：

- 支持创建、编辑、删除、查询项目。
- 项目名称唯一。
- 一个项目可以包含多个服务和多个环境。
- 删除项目时应保护已有服务、环境、发布记录等关联数据，不能静默丢失审计历史。

### 6.2 服务 Service

服务是可发布实体，属于一个项目。

需求：

- 服务在定义层面不绑定环境。
- 一个服务可以部署到多个环境。
- 服务名称在项目内应可明确识别。
- 服务应维护稳定的 `slug`，作为 `service_key`；`service_key` 只需在所属项目内唯一，不要求全局唯一。
- 服务应能被 Web、API 和 Agent 以名称或 ID 查询。
- 服务本身不保存“当前版本”；运行版本应优先通过服务器级部署状态推导，环境级当前版本只是聚合视图。

### 6.3 服务版本 ServiceVersion

服务版本来自 CI/CD 注册或管理员维护，是发布系统选择和审计的核心对象。

需求：

- 同一服务下版本号唯一。
- 系统必须保留服务版本历史，不只保存最新版本。
- 版本属于服务，不属于环境。
- 版本至少包含版本号、登记时间和来源。手动登记可以只填写版本号；外部 CI 在构建产物可用后按需补充 commit、制品和外部运行信息，不同步失败构建状态。
- 可选保存 commit SHA、分支、外部构建链接、制品链接和 metadata；平台专属字段不进入 `ServiceVersion` 的通用字段。
- 本期 `source` 只由服务端写入 `manual` 或 `ci`：管理员手动登记固定为 `manual`，统一外部登记接口固定为 `ci`；客户端不能指定来源。历史 `api` 值不作为新接口输出。
- 制品要求由部署目标决定：后端 OCI 镜像目标要求不可变 digest 形式的 `artifact_url`；包下载或脚本内解析版本等目标可以只依赖版本号或使用其适配的制品形式。
- 发布单必须选择一个明确版本。
- 发布单建议保存 `service_version_id`，同时冗余保存版本号文本快照，确保历史审计稳定。
- 不应物理删除已经被发布单或发布记录引用的版本。
- 页面和 Agent 查询可以默认只展示最近 N 个版本，但底层应保留完整历史。

版本关系：

```text
Service 1 -> N ServiceVersion

order-api:
  - 1.0.0
  - 1.1.0
  - 1.2.0

test 当前运行版本：由服务器级部署状态聚合得出
prod 当前运行版本：由服务器级部署状态聚合得出
```

### 6.4 环境 Environment

环境属于项目，用于区分 dev、test、staging、prod 等发布目标。

需求：

- 同一项目内 `tag` 唯一。
- 环境应显式保存是否为生产环境，例如 `is_production`；`tag` 只作为可读标识，不应通过 `tag == "prod"` 推导生产约束。
- 可选保存环境关键级别，例如 dev/test/staging/prod 或自定义等级，用于展示。
- 生产环境默认应用更严格约束，第一版生产发布必须由管理员确认。
- 环境保存 `release_frozen`，作为该环境唯一的发布冻结开关，并关联部署目标。
- 环境不直接保存某个服务的版本，环境当前状态应从服务器级部署状态聚合得出。
- 如果同一服务在同一环境的不同服务器运行版本不一致，环境级当前版本应展示为“混合版本”，并引导用户查看服务器级版本明细。

### 6.5 基础设施 Server / Gateway / ServerGroup

服务器、网关和服务器组是基础设施资源。它们不是项目的核心下级对象，也不直接绑定某个服务。

需求：

- 服务器支持直连或通过网关访问。
- 服务器包含主机、端口、用户名、认证方式、部署脚本路径和状态。
- 服务器认证方式必须支持账号密码登录和 SSH 密钥登录。
- 密码和私钥等敏感信息必须加密或以安全方式保存，列表和日志中不得明文展示。
- 服务器连接配置应支持连接超时时间和命令执行超时时间。
- 服务器可以承载多个服务，一个服务也可以发布到多台服务器。
- 网关包含主机、端口、用户名、区域、状态和说明。
- 服务器组用于把多台服务器组织成一个可复用运行目标。
- 服务器和网关配置属于高风险配置，默认仅管理员可维护。

敏感凭据管理要求：

- SSH 密码、私钥、私钥 passphrase 等敏感字段必须加密存储。
- 加密密钥应通过环境变量或挂载文件提供，例如 `APP_ENCRYPTION_KEY`，不得写入数据库或镜像。
- 列表页和详情页不得回显完整敏感值，只允许显示认证方式、凭据是否已配置、最后更新时间等摘要信息。
- 日志、事件、错误信息和 API 响应必须统一脱敏。
- 应支持管理员重置服务器凭据；重置后旧凭据不可再读取。
- 应预留加密密钥轮换方案，至少保证后续可以通过离线或受控任务重新加密敏感字段。
- 备份和恢复文档必须提醒同时备份或重新配置加密密钥，否则加密凭据无法解密。

### 6.6 部署目标 DeploymentTarget

部署目标是发布系统最重要的配置对象，用来描述“某个服务在某个环境应如何发布到哪里”。

`DeploymentTarget` 连接：

- 项目。
- 服务。
- 环境。
- 执行器类型。
- 运行目标。
- 执行配置。

需求：

- 同一服务和环境应能唯一解析到一个启用的部署目标，避免发布时歧义。
- 部署目标包含执行器类型，例如 ssh、mock、webhook、jenkins、kubernetes。
- 部署目标包含显式 `artifact_type`，用于声明执行时对版本制品的要求；本期取值为 `version_only`（脚本可自行按版本解析）和 `oci_image`（必须使用 OCI digest 制品）。字段为非空且默认 `version_only`，保证既有部署目标继续沿用现有脚本行为；该字段不是从 `script_path` 推断。
- 部署目标包含执行器配置，例如脚本路径、环境变量、Webhook URL、Job 名称、namespace 等。
- 部署目标包含运行目标，例如服务器、服务器组、K8s namespace、Webhook endpoint 或 Jenkins job。
- 第一版只实现服务器和服务器组作为运行目标，但模型不应锁死在服务器模式。
- 第一版内置执行器只支持 Mock/Dry-run 和 SSH。
- 支持多服务器顺序、并行策略和权重等后续扩展。
- 发布前必须校验目标服务和环境存在可用部署目标。
- 发布执行时必须基于当时的部署目标生成执行快照，避免后续修改部署目标影响历史发布追溯。
- 执行快照应在 `DeployRecord` 创建时生成，冻结部署目标、目标服务器列表、网关摘要、脚本路径、执行器配置摘要、服务版本和系统注入变量等结构性信息。
- 发布执行时必须把发布单选择的 `ServiceVersion` 注入执行上下文，避免部署脚本无法知道要发布哪个版本。

版本与制品传递边界：

- 第一版由发布系统负责选择、记录和注入版本信息。
- 第一版不强制发布系统直接下载、解压或校验制品；SSH 脚本可以根据注入变量自行拉取制品。
- 如果 `ServiceVersion` 包含制品地址，系统应把地址注入执行器上下文。`artifact_type=version_only` 缺少制品地址时给出 warning；`artifact_type=oci_image` 缺少制品地址或不是 digest 引用时必须 block。
- 标准注入变量建议包含 `RELEASE_VERSION`、`SERVICE_VERSION_ID`、`ARTIFACT_URL`、`COMMIT_SHA`、`RELEASE_REQUEST_ID`、`DEPLOY_RECORD_ID`、`PROJECT_KEY`、`SERVICE_KEY`、`ENVIRONMENT_TAG`。
- `ARTIFACT_URL` 可能包含预签名 token 或临时凭据，应按敏感字段处理，日志、事件、快照展示和 API 响应中必须脱敏。
- 部署目标自定义环境变量不得覆盖系统保留变量；冲突时 preflight 应阻断或要求管理员修改配置。

执行快照应至少包含：

- 部署目标 ID 和名称。
- 执行器类型。
- 执行器配置摘要。
- 运行目标类型。
- 服务器或服务器组解析后的服务器列表。
- 经网关或跳板机访问时的网关配置摘要。
- 服务器执行顺序。
- 部署脚本路径。
- 关键环境变量或配置摘要，包括系统注入的版本、制品、发布单和发布记录标识。
- 快照生成时间。

示例：

```text
order-api + test
  -> executor: ssh
  -> runtime target: test-app-01, test-app-02
  -> config: /data/order-api/deploy.sh
  -> env: RELEASE_VERSION=1.2.0, ARTIFACT_URL=https://artifact.example/order-api-1.2.0.tar.gz

order-api + prod
  -> executor: ssh
  -> runtime target: prod-app-group
  -> config: /data/order-api/deploy.sh
```

### 6.7 SSH 执行器与连接配置

SSH 执行器是第一版真实发布能力的核心执行器，依赖服务器连接信息、认证凭据和部署脚本配置。

服务器连接配置需求：

- 支持主机地址、端口、用户名。
- 支持账号密码登录。
- 支持 SSH 密钥登录。
- 支持可选网关/跳板机连接。
- 支持连接超时时间。
- 支持命令执行超时时间。
- 支持测试连接。
- 支持启用、禁用、维护中等状态。

SSH 执行器异常处理需求：

- 连接超时应给出明确错误，例如“连接服务器超时”。
- 主机不可达应给出明确错误，例如“无法连接目标主机”。
- 登录失败应区分账号密码错误、密钥不可用、权限不足等常见原因。
- 部署脚本不存在或不可执行时，应明确提示脚本路径和服务器。
- 命令执行超时应标记为执行失败，并记录超时信息。
- 单台服务器失败时，应记录对应服务器日志，不应只给出笼统发布失败。
- 错误信息应便于员工和管理员定位问题，但不得泄漏密码、私钥、token 等敏感信息。

SSH 命令安全要求：

- 部署脚本路径必须来自管理员维护的部署目标或服务器配置，普通发布申请不得临时传入任意命令。
- 执行命令应采用固定模板，例如调用指定脚本并注入环境变量，避免拼接用户输入形成任意 shell 片段。
- 版本、制品、服务、环境、发布单和发布记录等动态信息应通过系统注入环境变量传入，不通过拼接命令行参数传入。
- 如确需自定义参数，只能由管理员在部署目标中配置白名单参数，发布单不得临时传入任意脚本参数。
- 不允许使用 `bash -c "..."` 等方式把发布单输入、自由文本或未校验变量拼接进远程命令字符串。
- 脚本路径、环境变量名和值应做格式校验和转义；敏感变量不得出现在可见日志中。
- SSH 执行器不得把发布说明、变更摘要等自由文本直接拼接进远程命令。
- 部署脚本必须具备幂等性或可重入性；同一版本重复执行、超时后重新发布、`partial` 后全量重放到已成功服务器时，不应产生破坏性副作用。
- 发布系统不保证部署脚本的业务幂等，脚本作者应负责脚本侧锁、重复执行检测和中间状态恢复；推荐使用锁文件、当前版本标记文件或等价机制。
- SSH 命令超时不保证远端进程一定被终止；接入方脚本应自行处理进程锁、重复执行和中间状态恢复。

### 6.8 发布单 ReleaseRequest

发布单是发布系统内部的执行意图对象。它不是 OA/SOP 审批单，也不是传统重流程工单，而是一次发版执行所需的最小上下文和状态载体。

需求：

- 发布单必须关联项目、服务、环境、服务版本、申请人。
- 发布单创建时默认只强制要求服务、环境、版本和调用身份。
- 发布单应支持保存简短发布说明，但非生产发布不应强制填写。
- 标题应优先由系统根据服务、环境和版本自动生成。
- 发布单可选保存外部单号或链接，例如 OA 单、SOP 文档、变更单或聊天记录链接；这些外部引用不是系统运行前提。
- 发布单可选保存变更摘要、回滚说明、验证说明和风险说明；第一版不把这些字段作为必填项或 preflight 阻断项。
- 发布单应保存调用来源：web、api、ai_agent、ci。`source=ai_agent` 应由 Agent API 服务端写入，不接受普通客户端通过通用创建发布单接口自行声明，避免来源伪造。
- 发布单应保存动作主体、授权用户、服务账号和最小来源元数据。Agent 元数据只用于审计和排查，建议包含 agent 名称、skill 版本、用户确认后的发布意图摘要、客户端请求 ID 或外部会话引用；不得把完整对话、prompt、模型推理过程或 token 明细作为发布单常规字段。
- 发布单应保存 preflight 状态和检查结果。
- 发布单可关联一个真实发布记录。
- 发布单应保存执行门禁状态和最终结果摘要，不直接保存服务器级执行细节。
- 回滚发布单应通过 `rollback_of` 关联原发布单。
- Agent 或 API 创建发布单必须支持幂等键。

幂等语义：

- 幂等键作用范围至少包含调用身份、业务动作、目标服务、目标环境和目标版本。
- 同一作用范围内重复提交相同幂等键，应返回第一次创建的发布单或发布记录，而不是创建重复发布。
- 如果同一幂等键对应的请求内容发生关键字段变化，应拒绝请求并返回冲突错误。
- 幂等键应有保留期限；第一版建议至少保留 24 小时，具体 TTL 可在 API 设计阶段确定。
- 幂等键在其作用域内必须有数据库唯一约束；并发冲突时应回查已存在记录并返回首次请求结果。
- Agent、CI/CD 或其他自动化调用确认、取消、创建回滚等状态变更接口时，也应支持幂等键或等价的重复提交保护。

### 6.9 发布记录 DeployRecord

发布记录是真实执行对象。

需求：

- 发布记录只在发布单获得执行资格后创建。
- 发布记录应关联项目、服务、环境、服务版本、发布单和部署目标。
- 发布记录应保存队列、运行、成功、失败、部分成功等执行状态。
- 发布记录应保存总服务器数、成功服务器数、失败服务器数。
- 发布记录应保存执行器名称、外部任务 ID、幂等键和执行尝试次数。
- 发布记录状态是执行状态的权威来源，发布单终态应由发布记录聚合回写。
- 发布记录应保存部署目标执行快照，确保管理员后续修改部署目标、服务器组或脚本路径后，仍可还原当次发布实际执行配置。
- 发布记录应保存执行占用、Worker 标识和最近心跳时间，用于表达当前执行者和识别僵尸任务。

### 6.10 服务器发布日志 ServerDeployLog

服务器发布日志记录每台服务器的执行状态和输出。

需求：

- 每个发布记录可以有多个服务器日志。
- 日志状态至少包含 queued、running、success、failed。
- 多服务器 fail-fast 导致未执行的服务器日志应标记为 `skipped`，避免长期停留在 queued。
- 日志应保存开始时间、结束时间、输出、错误信息和重试次数。
- 日志输出必须脱敏，不得泄漏密钥、token、SSH 私钥。

### 6.11 服务器部署状态 ServerDeploymentState

服务器部署状态用于表达“某个服务在某个环境的某台服务器当前运行哪个版本”，是环境级当前版本聚合的基础。

需求：

- 服务器部署状态应由成功的服务器级发布结果更新，不由发布单表直接推导。
- 状态至少关联项目、服务、环境、服务器、服务版本、发布记录和更新时间。
- 当发布记录为 `partial` 时，已经成功的服务器应更新到新版本，失败或未执行的服务器保持原版本。
- 环境级当前版本应从服务器部署状态聚合：全部服务器一致时显示该版本，不一致时显示“混合版本”。
- 服务器部署状态是运行视图，不替代发布记录、服务器日志和审计事件。

### 6.12 发布保护

第一版只提供固定的生产保护和环境冻结，不提供通用发布规则引擎。

需求：

- 非生产环境由发起人本人确认；生产环境必须管理员确认，二者均不可配置覆盖。
- 环境支持独立的手动冻结开关，用于客户环境、维护窗口等禁止随意发版的场景。
- 环境冻结后，preflight 和新建/确认发布被阻断；已 queued 的任务暂停领取，running 任务继续。
- 外部单号、SOP 链接、回滚说明和验证说明都是可选上下文，不作为第一版 preflight 阻断项。
- 同服务同环境已有运行中发布时，应默认阻断新的真实执行。
- 不支持系统级或服务级策略、策略优先级、策略合并和定时冻结；真实需求出现后再单独扩展。

### 6.13 发布事件 ReleaseEvent

事件流是审计核心。

需求：

- 发布单创建、preflight、发布保护判断、确认、驳回、取消、回滚申请、发布开始、执行器调用、单节点结果、发布结束都应记录事件。
- 事件必须记录时间、动作、备注、来源、动作主体、授权用户、服务账号和元数据。
- 审计事件应能区分“谁发起”“谁授权”“使用了哪个凭据”“从哪里调用”。
- 第一版审计重点是关键动作可追溯和可查询，不设计不可篡改账本、哈希链或外部审计存储。
- 如果需要修正错误事件，可通过管理员备注或补充事件表达，不要求实现严格只追加机制。

### 6.14 API Key / Service Account

API Key 是自动化调用凭据。

需求：

- API Key 明文只在创建时返回一次，数据库只保存 hash。
- API Key 支持名称、前缀、scope、所有者、过期时间、启用状态和最后使用时间。
- API Key 权限只按 scope 授权，不因创建者是管理员而自动拥有全部权限。
- API Key 必须支持禁用和删除；禁用后新的请求立即失效。
- API Key 不得绕过已配置的发布门禁。

第一版最小 scope：

| Scope | 说明 |
|-------|------|
| `release:read` | 读取项目、服务、版本、环境、发布单、发布记录和日志。 |
| `release:create` | 创建发布单和执行发布 preflight，不代表绕过确认直接执行。 |
| `release:confirm` | 确认、驳回、取消发布单；是否可确认生产发布由环境和调用身份决定。 |
| `release:rollback` | 创建回滚发布单；不代表绕过发布保护直接执行回滚。 |
| `deploy:read` | 读取执行状态、服务器日志和发布结果。 |
| `admin:write` | 管理用户、基础设施、部署目标、环境发布保护、API Key 等高风险配置。 |

说明：

- `admin:write` 不应自动绕过发布单、preflight、执行门禁和审计。
- API Key 即使具备 `release:confirm`，也不能绕过生产发布的管理员确认。
- 后续 AI Agent 接入时可以在此基础上细化 Agent 专用 scope。

### 6.15 执行器 contract

执行器 contract 是发布系统调用不同执行方式的最小语义约束。第一版不要求设计复杂插件系统，但 Mock/Dry-run 和 SSH 必须遵守同一契约。

输入上下文至少包含：

- 发布单、发布记录、项目、服务、环境和服务版本快照。
- 部署目标执行快照。
- 单台服务器或当前运行目标。
- 系统注入环境变量和部署目标自定义环境变量。
- 调用超时时间、幂等键和执行尝试次数。

执行语义：

- 第一版执行器调用按同步阻塞语义实现：Worker 调用执行器并等待单台服务器执行完成或超时。
- 执行器必须按服务器返回结果，不能只返回整单成功或失败。
- 执行器应尽量流式采集日志；如果第一版实现成本过高，至少必须分段或最终回写服务器日志，并记录开始、结束、退出码和错误摘要。
- 执行器必须把连接失败、认证失败、脚本不存在、权限不足、命令超时、退出码非 0 等错误归类。
- 执行器返回结果不得包含未脱敏的密码、私钥、token 或其他敏感值。
- 当前 PRD 不设计运行中紧急停止能力；运行中异常依赖命令超时、僵尸任务识别和管理员人工处理兜底。

返回结果至少包含：

- `status`：success 或 failed。
- `exit_code`：如执行方式支持，应记录远端命令退出码。
- `started_at`、`finished_at`、`duration_ms`。
- `log_output` 或日志片段引用。
- `error_code`、`error_message` 和脱敏后的错误摘要。
- 外部执行系统任务 ID；SSH 执行器可为空。

## 7. 核心流程需求

### 7.1 Web 提交发布单

流程：

```text
选择服务/环境/版本
  -> 执行 preflight
  -> 按策略补充最小必要信息
  -> 创建发布单
  -> 本人确认或等待管理员确认
  -> 进入发布队列
  -> 执行器执行
  -> 查看发布记录和日志
```

需求：

- 提交页面必须阻止明显缺失的服务、环境、版本。
- 普通发布默认不要求填写发布原因、风险说明、验证说明和回滚说明。
- 非生产发布由执行人本人确认，生产发布由管理员确认。
- preflight 为 block 时不能创建或确认真实发布。
- warning 可以继续，但必须可见。
- 创建成功后应引导用户查看发布中心或发布记录。

### 7.2 发布确认与执行门禁

流程：

```text
系统根据策略判断门禁模式
  -> 非生产 self_confirm：执行人本人确认
  -> 生产 admin_confirm：等待管理员确认
  -> 确认通过后创建发布记录并入队
```

需求：

- 普通员工可以执行或确认策略允许的发布。
- 生产发布必须由管理员确认。
- 任何本人确认、管理员确认和驳回都必须留下事件。
- 驳回必须保存驳回原因。
- 确认时应再次校验关键 preflight 或策略，避免提交后环境变化导致错误发布。

### 7.3 发布队列与执行

流程：

```text
确认通过或策略允许直接执行
  -> 创建 DeployRecord
  -> 创建 ServerDeployLog(queued)
  -> 调度器检查服务器占用
  -> 领取可执行任务
  -> 调用执行器
  -> 回写服务器日志和发布记录
  -> 回写发布单终态
```

需求：

- 第一版发布队列的并发控制粒度以服务器为主。
- 同一服务器同一时间只能有一个发布进入 running。
- 不同发布记录如果目标服务器互不重叠，可以并行执行。
- 一个发布单部署到多台服务器时，应在目标服务器都可用时整体领取。
- 第一版不支持分批领取、抢占或复杂优先级调度；队列公平性优先采用创建时间排序等简单策略。
- 第一版按单服务实例和内置 Worker 设计，不考虑多实例 Worker 抢占任务。
- Worker 领取任务时应写入 Worker 标识、租约过期时间和心跳时间；任务执行期间必须定期续租。
- 租约 TTL 和心跳超时必须基于对应服务器的命令执行超时时间与心跳周期派生，不得小于命令执行超时时间，避免合法长任务被误判为僵尸任务后重复领取。
- 如果发布记录涉及多台服务器，领取时必须在同一事务中确认所有目标服务器租约可用，否则保持 queued 等待后续调度。
- Worker 领取 queued 任务时，必须在同一事务中确认发布单和发布记录仍处于可领取状态；如果用户已取消，则不得进入 running。
- 取消 queued 发布单时，也必须在同一事务中确认该发布尚未被 Worker 领取；如果已进入 running，应拒绝取消并返回当前运行状态。
- 多服务器发布第一版默认按顺序执行并采用 fail-fast：某台服务器失败后停止后续未开始服务器，已成功服务器不自动回退。
- 发布队列应保留基本公平性，长期等待的多服务器发布不能被持续到来的单服务器发布无限饿死；具体优先级策略可在技术方案阶段设计。
- 系统重启后应能恢复 queued 任务。
- 运行中的僵尸任务应通过租约过期或心跳超时识别，并标记失败或待人工处理。
- 执行器调用必须支持幂等保护。
- 第一版取消只覆盖未进入执行前的发布单；进入 running 后不提供系统级紧急停止入口，应在页面提示管理员结合执行器、服务器和超时机制人工处理。

重试与重新发布语义：

- 第一版不支持把 `failed` 或 `partial` 的发布记录直接改回 `queued/running`。
- `DeployRecord.attempt_count` 只用于同一服务器执行过程中的受控重试或执行器内部重试统计，不表达发布单级重新发布。
- 用户看到的“重新发布”应创建新的发布单，并重新执行 preflight、策略、确认和审计流程。
- 重新发布同一版本应使用新的幂等键；如果复用旧幂等键，应命中旧发布单而不是创建新的发布。

### 7.4 回滚发布单

流程：

```text
选择原成功或部分成功发布单
  -> 选择或推荐回滚版本
  -> 创建回滚发布单
  -> 执行回滚 preflight
  -> 按环境发布保护确认
  -> 入队执行
```

需求：

- 回滚必须创建新发布单，不能直接执行。
- 回滚发布单必须关联原发布单。
- 生产回滚仍需遵守生产管理员确认和环境冻结门禁。
- 默认推荐回滚版本为目标环境中该服务最近一次成功发布且不同于当前版本的版本。
- 如果找不到最近成功版本，系统不应自动猜测，应提示用户手动选择可回滚版本并补充风险说明。
- 首次部署失败且没有可回滚版本时，回滚入口应明确提示“暂无可回滚版本”，引导用户重新发布修复版本或人工处理，不展示无效回滚动作。
- 回滚 preflight 至少检查原发布单是否成功或部分成功、回滚版本是否存在、目标环境是否已有运行中发布。

### 7.5 部分成功后的恢复

`partial` 是真实生产中需要优先补救的状态，第一版必须给出可执行恢复路径，但不把它扩展成复杂编排能力。

恢复原则：

- `partial` 发布单在治理摘要上按失败处理，但服务器部署状态必须保留已成功服务器的新版本。
- 发布单详情应明确展示成功服务器、失败服务器、未执行服务器和环境级“混合版本”状态。
- 从 `partial` 恢复必须创建新的发布单，不能直接修改原发布记录状态。
- 第一版默认恢复动作是全量重放到同一部署目标：重新发布同一版本，或回滚到推荐的上一个成功版本。
- 全量重放可能对已成功服务器重复执行，因此第一版依赖部署脚本幂等来控制副作用；这是为降低定向修复复杂度接受的 MVP 取舍。
- 第一版不强制支持“只补失败服务器”的定向修复；如后续支持，必须作为明确的部署目标执行范围能力设计，并纳入 preflight、审计和执行快照。
- 恢复发布完成后，环境级当前版本应重新从服务器部署状态聚合；全部服务器一致时退出“混合版本”状态。

### 7.6 CI/CD 版本注册

流程：

```text
外部 CI 构建产物可用
  -> 调用统一版本登记接口
  -> 系统按 project_key + service_key 解析既有服务
  -> 创建或幂等返回服务版本
  -> 后续发布选择该版本
```

需求：

- 提供统一接口 `POST /api/v1/version-registrations`；所有外部 CI 和服务调用同一地址。
- 外部请求必须包含 `project_key`、`service_key`、`version`；可选包含 `commit_sha`、`artifact_url` 和 metadata。GitLab Pipeline、Jenkins Job、GitHub Actions Run 等平台专属信息写入 metadata，不形成通用必填字段。
- HTTP 请求中的 `metadata` 使用 JSON 对象；服务端校验后序列化为 `ServiceVersion.metadata` 的 JSON 文本。登记接口不校验 `artifact_url` 的具体制品格式，格式与必填校验在目标 preflight 按 `artifact_type` 执行。
- `project_key` 对应 `Project.slug`，`service_key` 对应该项目下的 `Service.slug`。二者共同定位服务，避免把非全局唯一的服务 key 当作全局 ID。
- 项目和服务必须预先由管理员维护；CI 只能登记版本，不自动创建项目或服务。
- 第一版不维护 repository 实体或 repository 到服务的映射。外部 CI 项目只需配置一次 `AI_PUB_PROJECT_KEY` 受保护变量；单仓多服务 Job 传递自己正在构建的模块 `service_key`。
- 外部登记必须带 `Idempotency-Key: {provider}:{run_id}`。`service_versions` 保存可空的 `registration_idempotency_key` 与请求指纹，并对 `(service_id, registration_idempotency_key)` 建唯一索引。请求指纹只由 `version`、`commit_sha`、`artifact_url` 的规范化值构成，排除 metadata、构建时间和运行链接等可变追溯信息：同服务同 key 且指纹相同返回已有版本 `200`，同 key 指纹不同返回 `409 idempotency_conflict`。
- 同一服务下版本号必须有数据库唯一约束。不同幂等键登记相同版本时，commit、制品一致返回已有版本，二者不一致返回 `409 version_conflict`，不能覆盖历史版本。
- 管理员手动登记不使用幂等键；同一服务重复版本一律返回 `409 version_conflict`，不覆盖既有版本。手动与 CI 共用相同的版本唯一性和冲突判断。
- 外部 CI 在制品准备完成后登记版本，不创建、确认或执行发布单。
- 接口必须使用具备 `version:write` scope 的 API Key。即使仅部署在内网也不能省略鉴权：该接口会登记后续可部署的制品；可使用一个 GitLab Group 级受保护 API Key，避免为每个服务维护 Key。
- 外部系统的 repository 地址或运行链接可写入版本 metadata 供审计与跳转，但本期不用于服务定位。
- 版本登记审计使用独立 `service_version_events`，关联 `service_version_id`，记录 `version_registered`、`actor_type(user/api_key/system)`、actor、API Key、幂等键和 metadata；不复用以发布单为中心的 `release_events`。
- GitLab CI 的 OCI 镜像登记是首个接入示例；其他 CI 复用统一接口，不需要平台专用版本模型。签名校验、IP 白名单、OIDC 与 repository 映射作为后续增强项讨论。

## 9. Preflight 需求

Preflight 是发布前的风险检查系统。

检查结果：

- `pass`：允许继续。
- `warning`：允许继续，但需要展示风险。
- `block`：禁止继续。

检查项结构：

- `code`
- `status`
- `message`
- `severity`
- `metadata`
- `checked_at`

第一版必须覆盖：

- 服务存在。
- 环境存在。
- 版本存在。
- 服务和环境属于同一项目。
- 服务在目标环境存在部署目标。
- SSH 发布目标的服务器连接配置完整。
- 发布版本具备执行器所需的版本号、commit 或制品地址；缺失时按部署目标要求给出 warning 或 block。
- SSH 发布目标存在最近一次测试连接结果；如果最近测试失败或过期，应给出明确提示。
- 是否在发布 preflight 中实时测试 SSH 连接应由策略控制。第一版默认可使用最近一次测试结果，生产环境可配置为强制实时测试。
- 生产环境判断必须使用环境显式字段，例如 `is_production`，不得依赖环境 tag 文本。
- 回滚发布单的原发布单合法。
- 回滚目标版本存在。
- 当前调用方有创建、执行或确认权限。
- 策略没有冻结。
- 同服务同环境没有运行中发布。

SSH 连接检查原则：

- 创建或更新服务器连接配置时，必须支持手动测试连接。
- 发布 preflight 必须检查连接配置完整性。
- 发布 preflight 不应无条件实时连接所有服务器，避免发布流程因临时网络抖动过慢或误阻断。
- 当策略要求实时测试时，连接失败应作为 `block`；否则最近测试失败或过期可作为 `warning` 或按策略升级为 `block`。

后续可扩展：

- CI 构建状态。
- 制品存在性。
- 监控告警状态。
- 执行器可用性。
- 定时冻结窗口。

## 10. 状态机需求

### 10.1 发布单状态

发布单状态表达执行门禁流程和最终结果摘要。进入执行后的 `queued`、`running`、`success`、`failed` 是由发布记录状态聚合回写的摘要，不是服务器级执行权威状态。

建议状态：

- `pending_confirm`：待确认或等待策略门禁满足。
- `queued`：已通过确认或策略允许直接执行，已创建发布记录，等待发布队列调度。
- `running`：关联发布记录执行中。
- `success`：关联发布记录聚合结果为成功。
- `failed`：关联发布记录聚合结果为失败或部分失败后判定为失败。
- `rejected`：已驳回或门禁不通过。
- `cancelled`：已取消。

状态流：

```text
pending_confirm -> rejected
pending_confirm -> cancelled
pending_confirm -> queued
queued -> cancelled
queued -> running
running -> success
running -> failed
```

说明：

- 第一版不新增确认逾期终态。
- 确认逾期可通过字段、事件、通知和列表视觉标记表达；第一版不因超期自动取消发布单。
- 取消仅允许在未进入执行前发生。
- `queued -> running` 的 Worker 领取与取消操作必须互斥；任一动作成功后，另一个动作必须基于最新状态失败并返回明确提示。
- `queued -> cancelled` 只允许在 Worker 尚未领取发布记录前发生。
- 发布单不得绕过发布记录直接进入 `running/success/failed`。
- 发布单详情页可以展示聚合状态，但执行细节以发布记录和服务器日志为准。

### 10.2 发布记录状态

建议状态：

- `queued`
- `running`
- `success`
- `failed`
- `skipped`
- `partial`

说明：

- `partial` 表示部分服务器成功，且存在失败或因 fail-fast 跳过的服务器。
- 发布记录状态必须能从服务器日志聚合得出。
- 发布记录状态变更后，应由应用服务统一回写发布单摘要状态。
- 第一版默认聚合规则：只要任一目标服务器失败或 skipped 且至少一台成功，发布记录状态为 `partial`；如果全部目标服务器失败或 skipped 且没有成功服务器，状态为 `failed`；如果全部成功，状态为 `success`。
- 第一版发布单摘要状态中，`partial` 应回写为 `failed`，并在摘要中保留部分成功计数，避免用户误以为发布整体成功。
- `partial` 场景下环境级当前版本不得继续展示为单一旧版本，应根据服务器部署状态展示为“混合版本”。
- 后续如需要按阈值判定成功，应作为明确策略能力设计，不在第一版隐式支持。

### 10.3 服务器日志状态

建议状态：

- `queued`
- `running`
- `success`
- `failed`
- `skipped`

### 11.1 业务 API

应稳定支持：

- 认证 API。

- 项目、服务、服务版本 API。
- 外部 CI 统一版本登记 API。
- 环境、基础设施、部署目标 API。
- 发布单 API。

- 环境发布保护 API。

### 11.2 发布单 API

必须支持：

### 11.3 Agent API

Agent API 是后置增强能力。第一版应先用 Web/API + React 最小管理界面跑通基础发布执行流程，待流程稳定后再接入 AI Agent。

后续 Agent 接入阶段必须支持：

- 查询服务候选。
- 查询服务版本。
- 查询部署目标。
- 查询环境候选。
- 发布意图 preflight。
- Agent 创建发布单。
- 查询发布单 summary。

Agent API 是业务 API 的薄封装，用于降低 AI 调用复杂度，不是第二套发布流程。

Agent API 创建发布单时应在服务端固定写入 `source=ai_agent`，并把必要来源信息写入 `ReleaseRequest.metadata` 或审计事件元数据。Agent API 仍复用同一个发布单应用服务、preflight、确认、队列、审计和权限校验，不新增独立工单、审批流或执行路径。

### 11.4 外部接入 API

新项目不要求兼容旧系统接口。外部接入 API 应围绕以下目标设计：

- CI/CD 可以注册版本和查询发布结果。
- 自动化工具可以创建发布单、查询发布单和查询发布日志。
- AI Agent 可以通过薄封装 API 降低误操作风险。
- 所有会触发真实发布的 API 都必须进入统一发布执行流程。
- API 应天然适合 OpenAPI 文档和未来 MCP/工具化封装。

## 13. 技术架构与数据库迁移要求

### 13.1 总体架构

第一版采用前后端分离架构：

- 后端使用 Go 实现，提供 REST API、后台调度、执行器调用、审计事件和数据库迁移能力。
- 前端使用现代 SPA 管理台，面向员工、管理员和发布处理场景。
- API 以 OpenAPI 作为文档和工具化基础，便于 Web 前端、CI/CD、AI Agent 和后续 MCP/工具封装调用。
- 应用容器负责提供 SPA 静态资源，并在启动时完成安全迁移、API 服务和 Worker 启动编排。

第一版不引入微服务拆分。发布系统核心域模型、API、队列调度和执行器应在一个后端服务内完成，降低部署和维护成本。

开发顺序：

1. 先完成 Go 后端核心能力和 REST API。
2. 配套 React 最小管理界面，优先支持人工方式跑通发布流程。
3. 在管理页面和基础执行流程稳定后，再接入 AI Agent。
4. AI Agent 接入阶段再补 Agent API、创建 skill，并测试 AI 调用发布流程。

### 13.2 后端技术架构

当前后端技术栈：

- Go。
- 标准 `net/http` 与 `ServeMux`。
- REST API。
- `database/sql`。
- 显式 migration。
- MySQL repository 与后续 PostgreSQL 方言隔离。
- 内置 Worker 处理发布队列和状态修复。

后端框架约束：

- 第一版保持贴近 Go 标准库，不引入重型 Web 框架，避免框架复杂度压过发布领域模型。
- API 层只负责认证、鉴权、参数校验和响应转换；发布单、preflight、确认、队列、执行器、审计等业务逻辑应放在应用服务和领域层。

当前后端分层：

```text
cmd/server
internal/httpapi        # REST API、middleware、auth、OpenAPI 适配
internal/app            # use case：创建发布单、确认、preflight、执行
internal/domain         # Project、Service、ReleaseRequest 等领域对象
internal/repository     # 显式 SQL 和事务封装
internal/executor       # Mock/Dry-run、SSH
internal/worker         # 发布队列调度、僵尸任务修复
internal/migration      # MySQL migration runner
internal/config         # 配置加载和校验
internal/auth           # 登录、密码哈希和会话 token
```

后端关键约束：

- 发布单状态流转必须集中在应用服务内，不散落在 API handler 和 Worker 中。
- 队列领取和服务器级并发控制必须有清晰事务边界。
- Worker 必须等待数据库迁移成功后启动。
- 执行器 contract 必须与发布单流程解耦，避免 SSH 细节污染发布单领域模型。

### 13.3 前端技术架构

第一版前端定位为 React 最小管理界面。它不是临时页面，而是正式前端架构的第一阶段，只覆盖人工跑通发布闭环所需的核心页面，后续再逐步扩展完整管理台能力。

当前前端技术栈：

- React。
- TypeScript。
- Vite。
- 手写轻量路由状态。
- 直接通过 `fetch` 调用后端 API。
- Ant Design。

选择理由：

- 该系统是登录后的管理台和发布操作台，不需要 SSR，Vite SPA 足够简单。
- 第一阶段不做完整管理台，避免过早投入复杂导航、统计、筛选和视觉细节。
- React + TypeScript 生态成熟，适合复杂表单、状态页、详情页和实时轮询。
- Ant Design 适合表格、表单、筛选、抽屉、确认弹窗、状态标签等后台管理场景，能降低第一版 UI 成本。

前端当前实现仍集中在 `web/src/App.tsx` 与 `web/src/styles.css`，适合 MVP 阶段快速闭环。后续如继续扩展，可按以下方向拆分：

```text
src/app              # 路由、Provider、全局初始化
src/pages            # 页面级组件
src/components       # 通用 UI 组件
src/features         # 发布单、服务、环境、服务器等领域功能
src/services         # API client
src/types            # API/领域类型
src/utils            # 时间、格式化、状态映射
```

前端关键约束：

- 页面围绕用户任务组织，不围绕数据库表机械堆页面。
- 发布单详情是核心页面，应能聚合 preflight、策略、确认、发布记录、服务器日志和事件流。
- 发布状态和日志查询应使用可控轮询，避免过度请求。
- 可见文案默认中文，产品名词保持统一。
- 不在前端保存任何敏感凭据明文。

### 13.4 数据库定位

第一版数据库策略：

- MySQL 8 是开发、自动化验收、演示和生产部署的唯一运行数据库。
- 本地开发和验收通过 Docker Compose 启动 MySQL，不要求开发者安装或暴露宿主机数据库。
- 不使用 ORM。数据库访问以显式 SQL 为主，并集中在 repository 层。
- 不使用重 ORM。数据库访问应以显式 SQL 为主，通过 repository 层封装。
- 基础 CRUD 可以使用轻量 SQL 工具辅助，但发布单状态流转、队列领取、并发控制、审计事件写入等关键路径必须使用明确 SQL 和事务。

不使用重 ORM 的原因：

- 发布系统关键路径包含状态机、队列领取、服务器级并发控制和审计写入，事务边界必须清楚。
- SQL 明确可读，更利于排查生产问题和 AI Agent 后续维护。
- 避免 ORM 隐式查询、级联删除、自动迁移等行为影响审计数据和发布流程。

### 13.5 MySQL 与 PostgreSQL 扩展边界

第一版只承诺 MySQL 8。项目开源后如需 PostgreSQL，应作为独立数据库方言接入，不提前维护第二套运行路径。

SQL 约束：

- 业务 SQL 必须集中在 repository 层，app、worker 和 HTTP 层不得包含数据库方言 SQL。
- 使用 `?` 占位符。
- MySQL 专属语法必须限于 MySQL repository 或 migration，不得泄漏到业务服务层。
- 避免数据库 JSON 函数，JSON 数据按文本保存，由应用层解析。
- 布尔值按 `0/1` 保存。
- 时间由应用层统一生成，建议存 UTC 时间。
- 避免依赖数据库自动更新时间。
- 避免存储过程和复杂视图。
- 不把核心业务逻辑写入数据库函数。

PostgreSQL 后续接入要求：

- 新增 `migrations/postgres`，不修改既有 MySQL migration。
- 新增 PostgreSQL 连接与 repository 适配；若占位符、锁或迁移 SQL 不兼容，应在 repository 边界隔离。
- 复用 Docker Compose 验收场景，验证 migration、Mock 发布闭环、事件流和队列语义。
- 未完成上述工作前，不在文档、配置或发布产物中宣称支持 PostgreSQL。

### 13.6 迁移版本管理

系统不使用 ORM AutoMigrate。所有库表变更必须通过显式 migration 文件完成。

迁移要求：

- migration 只能向前追加，不修改已发布 migration。
- 每个 migration 必须有版本号、up 脚本和必要时的 down 脚本。
- 当前维护 `migrations/mysql` 和 demo/local 用 `migrations/sqlite`；MySQL 8 是生产和正式集成验收数据库。
- 每次库表变更必须考虑旧数据兼容和历史审计数据保护。
- 发布单、发布记录、审计事件等历史表不得做危险级联删除。
- 删除字段应分阶段：先停止读写，再后续版本删除。
- 每次发布前应验证 MySQL 8 能从空库迁移到最新版本。

建议目录：

```text
migrations/
  mysql/
    000001_init.up.sql
    000001_init.down.sql
  sqlite/
    000001_init.up.sql
    000001_init.down.sql
```

### 13.7 容器化自动迁移

系统使用 Docker 容器化部署时，应减少进入容器手动执行迁移命令。

推荐启动流程：

```text
容器启动
  -> 读取 DB_DIALECT
  -> 按 mysql/sqlite 读取对应连接配置
  -> 获取迁移锁
  -> 检查 schema_migrations
  -> 校验 migration checksum
  -> 检查待执行 migration 分级和生产保护条件
  -> 执行允许自动执行的 migration
  -> 写入迁移结果
  -> 启动 Web/API/Worker
```

生产迁移要求：

- 应用启动时可以自动执行安全迁移。
- 生产环境必须有自动迁移开关。
- 生产环境自动迁移只允许执行 `safe` migration。
- `guarded` migration 在生产环境必须显式配置允许，并要求运维确认已完成备份。
- `manual` migration 在生产环境不得自动执行；检测到待执行 manual migration 时应用必须停止启动并输出明确原因。
- 生产环境不自动执行 down migration。
- 应提供迁移检查模式，用于在真正启动 API/Worker 前查看待执行 migration、分级、checksum 和风险提示。
- 生产迁移前应有备份提示；涉及数据回填、大表索引、字段类型变化等风险操作时必须要求备份确认。
- 迁移失败时应用不得继续启动 API 和 Worker。
- Worker 必须等待迁移成功后才能启动。
- 迁移日志必须清楚，但不得输出敏感信息。
- 已执行 migration 必须校验 checksum，发现历史 migration 被修改时应停止启动。

迁移记录表建议包含：

- version。
- dialect。
- checksum。
- applied_at。
- duration_ms。
- success。
- error_message。

### 13.8 迁移分级

迁移应分为：

- `safe`：允许自动执行。
- `guarded`：需要显式配置允许。
- `manual`：不允许自动执行。

`safe` 示例：

- 新增表。
- 新增 nullable 字段。
- 新增有默认值的字段。
- 新增非破坏性索引。
- 新增非破坏性约束。

`guarded` 示例：

- 大表加索引。
- 大量数据回填。
- 字段类型扩大。
- 数据格式转换。

`manual` 示例：

- 删除字段。
- 删除表。
- 重命名字段。
- 修改字段类型且可能丢数据。
- 拆表或合表。
- 删除或重写历史审计数据。

第一版应尽量只使用 `safe` migration。破坏性 migration 不得自动执行。

生产自动迁移默认策略：

- 默认允许自动执行 `safe` migration。
- 默认拒绝执行 `guarded` migration，除非显式开启并确认备份。
- 永远拒绝自动执行 `manual` migration。
- 任何被拒绝的 migration 都必须让应用停止启动，而不是跳过后继续运行。

### 13.9 数据库测试要求

最低测试要求：

- MySQL 8 migration 从空库执行成功。
- 基础 CRUD、发布单状态流转、队列领取、审计事件写入和查询在 MySQL 8 下通过。
- MySQL 8 下验证基础发布闭环可用，不要求高并发或多实例测试。
- 时间、布尔值、JSON 文本解析在 MySQL 8 下正确。

### 13.10 测试与风险管理

第一版测试应按风险分层：

- 单元测试覆盖状态机、环境发布保护、preflight 规则、幂等键判断和执行器结果聚合。
- 集成测试覆盖 repository、migration、审计事件、API 鉴权和 API Key scope。
- 功能测试覆盖服务器领取、Worker 心跳续租和僵尸任务修复。
- 端到端演示覆盖 Mock/Dry-run 完整发布闭环和 SSH 最小真实发布闭环。
- 前端检查覆盖核心表单、列表筛选、发布详情和错误状态展示。

开发任务清单应为关键验收项配套验证命令，例如后端测试、前端 lint/build、MySQL 8 容器 migration 和端到端 demo。

第一版风险登记至少包含：

| 风险 | 影响 | 缓解 |
|------|------|------|
| SSH 脚本差异过大 | 发布失败难定位 | 固定执行器 contract、注入标准变量、按服务器记录日志。 |
| SSH 超时未终止远端进程 | 重发或 partial 恢复时可能重复执行 | 要求部署脚本幂等、使用脚本侧锁或中间状态恢复机制。 |
| Worker 状态异常 | 任务长时间停留在 running | Worker 心跳、超时和僵尸任务修复。 |
| MySQL migration 或容器配置错误 | 开发和生产均无法启动 | MySQL 8 Compose 空库迁移和基础发布闭环测试。 |
| 审计事件缺失 | 无法追责 | 关键动作统一写事件，测试覆盖事件存在性。 |
| 凭据泄露 | 影响生产安全 | 加密存储、日志脱敏、敏感字段不回显、支持重置。 |
| 数据库迁移失败 | 应用无法启动 | 迁移分级、checksum、检查模式、失败停止启动。 |

## 14. 非功能需求

### 14.1 安全

- JWT 密钥、API Key、SSH 私钥、生产 token 不得打印到日志。
- API Key 只保存 hash。
- 高风险配置默认管理员可写。
- 生产发布必须由管理员确认。
- AI 和 service account 操作必须可追责。
- API、AI Agent 和 service account 调用应支持基础限流，避免误调用或异常集成拖垮系统。
- 敏感凭据应支持最小可行轮换路径：新增新密钥、用新密钥加密新数据、受控任务重加密旧数据、确认后停用旧密钥。

### 14.2 审计

- 所有关键动作必须写事件。
- 事件应结构化，便于后续查询和展示。
- 审计链路应能回答：谁发起、谁授权、使用哪个凭据、从哪里调用、做了什么、结果如何。
- 第一版不要求审计不可篡改，只要求事件记录可用于排查和追溯。

### 14.3 可靠性

- 发布执行必须有幂等保护。
- 系统重启后应恢复 queued 任务。
- 僵尸 running 任务应通过租约过期或心跳超时识别。
- SSH 连接超时、登录失败、脚本不存在、命令执行超时等错误必须被归类并回写到发布记录和服务器日志。
- 单台服务器执行失败不能覆盖其他服务器的执行结果，发布记录应能表达部分成功。
- 通知失败不能影响主流程。
- 数据库迁移必须可重复验证，失败时应用必须停止启动。

### 14.4 可运维性

- 提供健康检查。
- 提供运行摘要。
- 关键失败有可读错误。
- 发布日志按服务器保留。
- 执行器应支持测试或 dry-run。
- SSH 执行器应提供测试连接能力，并展示连接失败原因。
- 容器启动时的迁移结果应可在日志中明确查看。
- 提供 Prometheus 风格指标或等价指标接口，至少包含请求量、错误率、队列长度、运行中发布数、执行器失败数、通知失败数和 migration 结果。
- 后端日志应结构化，至少包含 request_id、actor、release_request_id、deploy_record_id 和 error_code。
- 后续可接入 OpenTelemetry 链路追踪；第一版不强制引入分布式追踪基础设施。

### 14.5 可扩展性

- 执行器可扩展。
- Preflight 检查项可扩展。
- 通知渠道可扩展。
- 外部上下文可扩展。
- 不为第一版引入复杂 BPM、完整 RBAC、多租户 SaaS。

### 14.6 易用性

- 新用户可以用 Mock/Dry-run 模式体验完整流程。
- 第一版开发者可以通过 Web/API 使用同一发布流程；后续 AI Agent 接入后继续复用同一流程。
- 错误信息应指导用户修复配置、版本或权限问题。
- 发布确认页面应适配移动端或窄屏视图，保证管理员或授权用户可以快速查看关键上下文并确认。
- 发布单详情页应按信息层级组织，例如基本信息、preflight、确认、执行记录、服务器日志和事件流分区或标签页展示，避免把所有信息堆在一个长页面中。

- README 首屏说明系统解决什么问题、适合谁、如何 5 分钟跑起来。
- 提供 demo 数据。
- 提供 Docker 或 Docker Compose 体验路径。
- 默认使用 Mock/Dry-run 执行器。
- 清理内部域名、镜像仓库、路径、账号和密钥。
- 提供 `.env.example`。
- 提供许可证、贡献指南和安全报告方式。
- 文档避免绑定单一公司流程。

## 16. 后续 AI Agent 能力

AI Agent 接入必须在 Web/API + React 最小管理界面已经跑通基础发布执行闭环之后进行。

后续 Agent 发版闭环：

```text
用户提出发布意图
  -> Agent 查询服务候选
  -> Agent 查询部署目标和版本
  -> Agent 执行 release intent preflight
  -> 用户确认摘要
  -> Agent 创建发布单
  -> 非生产本人确认或生产等待管理员确认
  -> Agent 轮询 summary 并汇报结果
```

后续 Agent 能力要求：

- Agent 查询接口应返回明确候选，不应在多匹配时替用户猜测。
- Agent 创建发布单前必须执行 preflight。
- Agent 创建发布单应由服务端自动写入 `source=ai_agent` 和最小审计元数据；元数据只记录 agent 名称、skill 版本、用户确认后的意图摘要、客户端请求 ID 或外部会话引用等必要信息。
- Agent 创建发布单必须支持幂等键。
- Agent 不能独立确认生产发布；生产发布必须等待管理员确认。
- Agent summary 接口应返回终端友好的状态、下一步和失败原因摘要。
- Agent 回滚发布单必须具备回滚 scope，并复用同一套回滚发布单流程。
- Agent 接入阶段不默认建设 Agent 注册中心、Agent 专用角色体系、完整对话留存或独立审批系统；如真实审计或合规需求出现，再作为独立能力扩展。

## 17. 新项目范围建议

### 17.1 第一版必须清晰设计的部分

- Go 后端分层架构。
- React 最小管理界面架构。
- 发布单领域服务。
- Preflight 服务。
- 环境发布保护。
- 发布队列调度。
- 执行器 contract。
- Mock/Dry-run Executor 与 SSH Executor。
- 审计事件模型。
- 数据库 migration 与容器启动自动迁移机制。
- React 最小管理界面的发布中心和发布记录体验。

### 17.2 第一版交付优先级

第一版实现时应按 MVP 优先级推进，避免所有需求同时进入必交付范围。

| 优先级 | 范围 | 说明 |
|--------|------|------|
| P0 | 发布单创建、本人/管理员确认、preflight、入队、执行、服务器日志、状态机、审计、基础 migration | 第一版闭环必须具备，缺失则系统不可用。 |
| P1 | 企业微信通知、发布中心筛选、基础 Dashboard、API Key、SSH 测试连接、冻结开关 | 提升可用性和运维效率，建议第一版包含，但可按排期取舍。 |
| P2 | 多级策略细化、Agent 字段和 API 预留、指标接口完善、更多执行器和通知渠道扩展点 | 不阻塞第一版人工执行闭环，可作为后续增强。 |

### 17.3 第一版可以保持简单的部分

- 用户认证。
- 项目、服务、服务版本、环境、基础设施基础 CRUD。
- API Key 管理。
- Dashboard 基础统计。
- 本地开发和演示环境。
- 单组织部署，不支持多租户。
- Agent API、AI skill 和 AI 调用测试放到基础执行流程跑通后的后置阶段。

### 17.4 应避免的产品方向

- 不要先做复杂 BPMN。
- 不要先做完整 RBAC。
- 不要做多租户。
- 不要做项目审批人、发布经理、环境负责人等复杂组织角色。
- 不要第一版引入微服务拆分。
- 不要第一版引入 SSR 或重型前端框架。
- 不要把 React 最小管理界面做成后续需要推倒重来的临时页面。
- 不要依赖 ORM AutoMigrate 管理生产表结构。
- 不要把 AI 做成绕过发布执行策略的新入口。
- 不要把发布系统做成 OA、SOP 或复杂审批流系统。
- 不要把任何具体执行器的细节散落在发布单流程里。
- 不要保留多个语义重复的发布入口。
- 不要为了通用化牺牲第一版可跑通的发布闭环。
- 不要第一版实现 test -> staging -> prod 的环境晋升编排；第一版每次发布以明确选择的服务、环境和版本为边界。
- 不要承诺一键安全回退被发布服务自身的数据库 schema 变更；回滚应用版本必须以数据兼容和迁移策略为前提。
- 当前 PRD 不考虑运行中发布的跨执行器紧急停止；先通过超时、租约、僵尸任务修复和人工介入兜底。

## 18. 第一版验收标准

第一版完成时，至少应满足：

- Web 创建发布单、确认、入队、执行、查看日志完整可用。
- Web/API 创建发布单走同一套 preflight 和策略。
- 普通发布不强制填写发布原因、风险说明、验证说明和回滚说明。
- 生产发布必须由管理员确认。
- 回滚创建新发布单，并关联原发布单。
- 发布队列以服务器为并发控制粒度。
- 同一服务器不会同时运行两个发布任务。
- 租约 TTL、心跳超时和命令执行超时关系明确，长任务不会被误判后重复领取。
- queued 发布的取消与 Worker 领取具备原子互斥，不会出现已取消但仍执行的状态。
- `ServiceVersion` 能通过标准变量注入 Mock/Dry-run 和 SSH 执行器上下文。
- `partial` 能按服务器日志聚合得出，并在发布单摘要中按失败处理。
- 多服务器 fail-fast 后，未执行服务器日志会标记为 `skipped`，不会长期停留在 queued。
- `partial` 后服务器级部署状态准确展示，环境级可展示“混合版本”，并可通过新发布单重新发布或回滚恢复。
- 冻结开关能阻断新发布和暂停 queued 任务领取，running 任务语义清晰可见。
- 事件流能完整追踪 Web、API、CI 和系统动作，并为后续 AI Agent 动作预留审计字段。
- 审计事件能支持关键动作查询、排查和追溯。
- Mock/Dry-run 执行器能跑通完整演示流程。
- SSH 执行器能跑通最小真实发布流程。
- SSH 执行器支持账号密码和密钥登录，连接超时、登录失败、脚本不存在、命令超时等错误能明确回写。
- MySQL 8 migration 能从空库迁移到最新版本。
- MySQL 8 能跑通完整发布闭环。
- 容器启动时能自动执行安全 migration；migration 失败时应用停止启动。
- 服务器级队列领取、租约、心跳和僵尸任务修复通过功能测试。
- Go 后端 API、Worker、migration runner 能在同一服务内完成第一版闭环。
- React 最小管理界面能完成发布单、发布记录、基础设施和环境发布保护的核心操作。
- 企业微信机器人 webhook 可用于生产待确认、发布失败和回滚申请通知，并保留后续渠道扩展能力。
- 基础运行指标和结构化日志可用于定位队列、执行器和通知问题。
- 后端核心测试、前端检查和端到端演示流程能通过。

## 19. 术语表

| 术语 | 含义 |
|------|------|
| `ReleaseRequest` | 发布单，表达一次发布执行意图、确认门禁、preflight、来源上下文和最终摘要。 |
| `DeployRecord` | 发布记录，表达一次真实执行及其队列、运行、成功、失败、部分成功状态。 |
| `ServerDeployLog` | 单台服务器的执行状态和日志。 |
| `ServerDeploymentState` | 某个服务在某个环境的某台服务器当前运行版本视图。 |
| `ServiceVersion` | 服务版本，来自 CI/CD 或管理员维护，是发布选择和审计对象。 |
| `ServiceVersionEvent` | 服务版本登记与版本动作审计，不关联发布单。 |
| `DeploymentTarget` | 服务 + 环境 + 执行器 + 运行目标的配置关系。 |
| 执行快照 | `DeployRecord` 创建时冻结的部署目标、服务器列表、脚本路径、环境变量和版本信息。 |
| preflight | 发布前检查，返回 pass、warning 或 block。 |
| 执行器 contract | 发布系统调用 Mock、SSH 或后续执行器时必须遵守的输入、输出和错误语义。 |
| 幂等键 | 自动化调用防重复创建发布单或发布记录的业务键。 |
| 租约 | Worker 领取服务器或任务后的临时占用记录，用于表达执行占用、心跳和超时。 |
| 僵尸任务 | Worker 已失联或长时间无心跳但仍处于 running 的任务。 |
