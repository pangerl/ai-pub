# AI Pub Agent 指南

## 工作原则

- 修改代码前先思考。对于会影响实现方式的重要假设应明确说明；歧义可能改变实现时先提问。优先采用满足需求的最简单方案。
- 保持改动精确且最小。不要重构、重排格式或删除无关代码；每一处改动都必须直接服务于当前请求。
- 对多步骤工作，先说明简短计划及验证标准，再在报告成功前完成验证闭环。
- 将已有的未提交改动视为用户的工作。不得覆盖或回退它们。

## 运行时与验证

- MySQL 8 是唯一受支持的运行时数据库；Docker Compose 是标准的本地集成环境。
- `make verify` 是代码级检查：运行 Go 测试、前端 lint 和生产构建。
- `make compose-check` 是集成验收检查：重新构建容器，从空 MySQL 数据库启动，并端到端验证发布流程。
- 如需清理后重新执行失败的 Compose 验证，使用 `make compose-down`；它会删除验证 profile 的资源和数据卷。

## 控制 token 与输出的验证方式

- 开发过程中使用范围最小的相关检查。例如，范围允许时只运行受影响的 Go 包测试，或只运行前端 lint/build，而不是完整测试套件。
- 不要在每次小改动后执行 `make compose-check`。涉及 migration、Compose 配置、repository/数据库行为、Worker、API 集成或发布流程时执行；在宣称跨服务改动或交付已经完整验收前也必须执行。
- 仅文档的改动通常只需运行相应的文档检查。纯前端展示改动通常先执行前端检查。
- 在不需要完整镜像重建时，可复用健康的 Compose 环境完成定向的手工/API 检查。`make compose-check` 应保留给其从干净环境进行验收的职责。
- 命令成功时，只报告命令、退出状态和简短成功标记；不要读取或复述完整的构建、容器日志。
- 命令失败时，先检查失败的服务或测试，并只读取相关日志的有限尾部。按需逐步扩展证据；不要一开始就读取所有 Compose 日志。
- 避免反复读取未变化的命令输出、生成文件或大文件。先总结结论，只有诊断失败时才重新打开细节。

## 完成标准

- 不得仅因编译通过就声称改动完成。应说明实际运行的检查及其结果。
- 改动跨越容器化运行时边界时，除非用户明确排除此项，完成标准必须包含相应的 Compose 验收结果。
